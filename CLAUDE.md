# CLAUDE.md

Android app (Kotlin, Jetpack Compose) for playing internet radio streams (HTTP/HTTPS, incl. HLS `.m3u8`) via ExoPlayer/Media3, with local station storage in Room and background playback.

## Project layout

- Single module: `app/`. Dependency versions live in `gradle/libs.versions.toml`; reference as `libs.foo.bar` / `alias(libs.plugins.foo)`, never hardcode a `group:artifact:version` string. Compose artifacts without a `version.ref` (`androidx-compose-ui`, `-material3`, etc.) are constrained by the `androidx.compose:compose-bom` platform instead — bump Compose via `composeBom`, not per-artifact.
- `app/src/main/java/com/urlradiodroid/`: `data/` (Room + repository), `ui/` (Compose screens + `ui/components`, `ui/playback`, `ui/theme`), `util/`.
- Lint: ktlint (`ktlintCheck` / `ktlintFormat`); `.editorconfig` allows PascalCase names on `@Composable` functions. CI: `.github/workflows/ci.yml` (test + Roborazzi verify, lint, `assembleDebug` on every push/PR), `release.yml` (tag `v*` → signed `assembleRelease` → GitHub Release).
- `./gradlew` locally (wrapper is committed, matches CI's Gradle version). `Makefile` wraps common invocations (`format`, `lint`, `test`, `build`, `check`, `install`/`run`) — keep in sync if a task name changes.
- Baseline Profile: `app/src/main/baselineProfiles/*.txt` (AGP 9.x reads this *directory*, not the old single `src/main/baseline-prof.txt` — that path silently produces an empty profile). Hand-curated: two wildcard rules covering `com.urlradiodroid.**`; verify with `./gradlew :app:expandReleaseArtProfileWildcards` and grep for `com/urlradiodroid`. No instrumented benchmark module (project has no `androidTest`), so re-curate by hand if the package structure changes shape.
- Release builds: `minifyEnabled true` + `shrinkResources true` (R8; `app/proguard-rules.pro` is the stock empty template, nothing project-specific needed). Signed via `signingConfigs.release`, reading `RELEASE_KEYSTORE_PATH`/`RELEASE_KEYSTORE_PASSWORD`/`RELEASE_KEY_ALIAS`/`RELEASE_KEY_PASSWORD` from env — if `RELEASE_KEYSTORE_PATH` is unset, signing is skipped and Gradle produces an unsigned APK (no debug-signing fallback). `release.yml` decodes `RELEASE_KEYSTORE_BASE64` to a workspace-local file, deleted with `if: always()`. `keystore/release.keystore` is gitignored, never committed — keep it and its password in a password manager; losing it means future releases can't use the same signing key.

## Architecture conventions

- `data/RadioStationRepository` is the only class touching `RadioStationDao`/`AppDatabase` directly — `ViewModel`s take a `RadioStationRepository`, never an `AppDatabase`/`Context`. Created via `RadioStationRepository.create(context)`.
- Two `ViewModel` styles: `MainViewModel`-style (plain `StateFlow`s + fire-and-forget `fun`s, optionally a `Channel`-backed `events` flow for one-shot feedback) for simple CRUD/search screens; `AddStationViewModel`-style (`UiState` data class + `Channel`-backed events) for screens with form validation. Pick based on screen complexity.
- **Full-row `@Update` trap**: `AddStationViewModel.save()` rebuilds a whole `RadioStation` from form fields, and `repository.updateStation()` is a full-row `@Update` — any field missing from the rebuilt object silently resets to its data-class default on edit. Every new `RadioStation` field must be added to `AddStationUiState` and threaded through *both* `save()` branches (new + edit), not just the entity. Bit us once with `isFavorite` (now removed); `sortOrder` and `genre` follow the correct pattern today.
- Deleting a station is immediate but undoable, not confirmation-gated: `MainViewModel.deleteStation()` deletes + reloads + emits `MainScreenEvent.StationDeleted`; `MainScreen` shows a Snackbar with Undo. `undoDelete()` re-inserts the same `RadioStation` (same `id`), which SQLite accepts as an explicit rowid, restoring original position (`getAllStations()` orders by `id ASC` within same-`sortOrder` rows).
- `RadioPlaybackService` (a `MediaLibraryService`, see Android Auto) exposes `playbackSnapshot: StateFlow<PlaybackSnapshot>` as the single source of truth for reactive UI — screens collect this instead of polling. Delegates to `playback/TimeshiftController` (buffer/seek for rewind-live) and `playback/PlaybackStateStore` (SharedPreferences; last-started station so `onStartCommand` can resume after the system kills+restarts the service with a null intent — this is what survives process death, not just backgrounding).
- Any `Activity` binding `RadioPlaybackService` and passing it into a `@Composable` (`MainActivity`, `PlaybackActivity`) must hold it as `mutableStateOf<RadioPlaybackService?>`, not a plain `var` — `onServiceConnected` fires after first composition, and a plain field mutation there is invisible to Compose.
- Reconnect-on-network-loss uses capped exponential backoff (`retryDelayMs`: 2s/4s/8s/16s, capped 30s, `MAX_RETRY_COUNT = 5`) — not infinite retry.
- `PlaybackActivity` opened from the mini player uses a custom slide-up-from-bottom transition (`res/anim/slide_up_in.xml`+`stay.xml`, via `ActivityOptionsCompat.makeCustomAnimation`); `finish()` is overridden to play the reverse (`slide_down_out.xml`, classic `overridePendingTransition` — not API-34's `overrideActivityTransition`, since minSdk is 29), covering every close path (back arrow, system back, programmatic `finish()`). Other entry points (notification, widget, App Shortcuts, Android Auto) skip the slide-up enter (no mini player to expand from) but keep the slide-down close.
- **Station icons** (`RadioStation.customIcon`: emoji string or image file path): default auto-generated via `util/EmojiGenerator`; Discover-added stations backfill a directory favicon async via `RadioBrowserApi.downloadFavicon()`/`IconStorage.saveImageBytes()`; overridable via `ui/components/StationIconPickerDialog.kt` (emoji picker, or system photo picker → `IconStorage.saveImage()` downscales to ≤512px JPEG in `filesDir/station_icons/`). A leading `/` is `IconStorage.isImagePath()`'s file-vs-emoji discriminator; every render site falls back to emoji if the file is missing. `NowPlayingBottomBar`'s mini-player cards deliberately don't show it (reclaims width for the station name). `AddStationViewModel` deletes only the *replaced* icon file, only once `save()` commits — no cleanup on cancel (accepted minor orphaned-file leak). **Not wired**: `PlaybackScreen` still shows only the auto-generated emoji, never `customIcon`. **Is wired**: notification large-icon via `RadioPlaybackService.currentCustomIcon` → `getCurrentLargeIcon()` (image path decodes via `IconStorage.decodeBitmap()`, non-path string = emoji override, `null` = auto-generated fallback).
- `ui/theme/Dimens.kt`: `Spacing` (4/8/16/24dp scale — only use where a value already matches exactly) and `AppShapes` (`extraSmall`=8dp icon controls, `medium`=16dp buttons/fields/chips, `large`=24dp primary cards, `extraLarge`=40dp pill mini-player cards), wired into `MaterialTheme(shapes = AppShapes)`. Reference via `MaterialTheme.shapes.large`/`Spacing.md`, not new `RoundedCornerShape(Ndp)`/raw `.dp` literals.
- **Landscape**: `ui/theme/WindowInfo.kt`'s `isLandscape()` (plain `LocalConfiguration.current.orientation` check — no `WindowSizeClass` dependency) is the orientation-detection seam. `PlaybackScreen` is the only screen with an actual orientation-branched layout: landscape swaps the vertical icon/name/controls stack for a two-pane `Row` (icon+name on the left, track title/play-button/sleep-timer on the right), mirroring Spotify/Apple Music's now-playing screen so nothing needs scrolling on a phone's shorter landscape height. `DiscoverStationsScreen`/`SettingsScreen`/`AlarmListScreen`/`AlarmEditScreen`'s card lists all cap at `widthIn(max = 600.dp)` + centered, matching `MainScreen`'s `StationItem` convention, so rows don't stretch edge-to-edge on landscape/wide screens. `AlarmEditScreen`'s time dialog falls back from the clock-dial `TimePicker` to the compact `TimeInput` below `TimePickerDialogDefaults.MinHeightForTimePicker` — the dial isn't scrollable inside a plain `AlertDialog` and gets clipped on a landscape phone's shorter screen height. `NowPlayingBottomBar`'s mini player needed no orientation changes — it's already a fixed 80dp height regardless of orientation.
- **Two-pane (tablet/foldable)**: `ui/theme/WindowInfo.kt`'s `isWideScreen()` (`LocalConfiguration.current.screenWidthDp >= 840`, Material's "expanded" width breakpoint) is width-based rather than orientation-based on purpose — a folded phone in landscape and an actual tablet/unfolded foldable can both report landscape orientation, but only the latter has 840dp+ to work with. `MainScreen` is the only screen using it: on a wide screen its `Scaffold` content branches to a `Row` of `StationListPane` (extracted composable holding the Discover chip/overflow-menu row, search field, and station list/empty-state — the exact content that fills the whole screen on a phone) at a fixed `360.dp` on the left, a 1dp `card_border` divider, and `PlaybackScreen`'s extracted `NowPlayingContent` composable (no `Scaffold`/`TopAppBar`/back button — those stay in `PlaybackScreen` for the phone-sized full-screen case) as a persistent detail pane on the right, showing `currentPlayingStation` or a "select a station" empty state if nothing's playing. The bottom mini player (`NowPlayingBottomBar`) is hidden on wide screens (`isWideScreen()` gates its `AnimatedVisibility` alongside the existing `currentPlayingStation != null` check) since the persistent detail pane already covers that role — tapping a station's play button still drives the same `currentPlayingStationId`/`onPlayStation` flow, it just renders in the second pane instead of pushing `PlaybackActivity`. Other screens (`DiscoverStationsScreen`, `SettingsScreen`, `AlarmListScreen`, `AlarmEditScreen`) are unaffected — still single-pane, capped at `widthIn(max = 600.dp)`.
- `URLRadioDroidTheme` is always-dark by design — no system-theme following, no light `ColorScheme`, no `dynamicColor`. Colors in `Color.kt` are hardcoded dark values; a default Material component without explicit `colors` would go invisible-on-light if theme-following were reintroduced. Don't reintroduce `isSystemInDarkTheme()` or a light `ColorScheme`.
- When a `Modifier` chain combines a rounded `shape` with `.clickable()`/ripple, the shape-clip must come before the click handler — otherwise ripple paints the full rectangular bounds past the rounded corners. Prefer `Card(onClick = ..., border = ...)` (shape-aware internally); for plain `Box`/`Row`, order `.clip(shape).clickable(...)`.
- `util/AppShortcuts.kt` publishes ≤2 dynamic App Shortcuts: last-played station (`PlaybackStateStore`) and first in manual list order, deduped if same. `AppShortcuts.refresh()` fires from `LaunchedEffect(allStations)` in `MainScreen`, so a shortcut can lag until the user reopens `MainActivity` (no dedicated observer — accepted). Both always show the auto-generated emoji, never `customIcon`. Intent targets `PlaybackActivity` with `EXTRA_AUTO_PLAY = true`, which **requires `PlaybackActivity` to be `android:exported="true"`** — any installed app can launch it with arbitrary `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` and start playback of an arbitrary URL. Accepted as low-severity (no data access/permission escalation), not gated behind a signature permission.
- **Wake-up alarms** (`ui/AlarmListScreen.kt`'s `AlarmActivity`): multiple daily alarms, each with enabled/time/station — not a full clock app (no day-of-week, no snooze). Built on `AlarmManager.setAlarmClock` (`ui/AlarmScheduler.kt`), not `setExactAndAllowWhileIdle`, since a user-visible alarm is documented exempt from `SCHEDULE_EXACT_ALARM`. That exemption isn't guaranteed on all devices — `setAlarmClock()` can still throw `SecurityException`, so `AlarmScheduler.schedule()` returns `Boolean` (checks `canScheduleExactAlarms()` first, wraps the call in `try/catch(SecurityException)`, never throws). `AlarmEditScreen`'s save is the only call site with UI to react to `false` (shows `alarm_permission_needed`, launches `Settings.ACTION_REQUEST_SCHEDULE_EXACT_ALARM`); `BootReceiver`/`AlarmReceiver` silently discard `false`.
  - Each alarm is a `data/WakeAlarm` Room row (schema v9, `AppDatabase.MIGRATION_8_9`) via `data/AlarmRepository` (`create(context)`, wraps `WakeAlarmDao`). Each row gets its own `PendingIntent` keyed by `AlarmScheduler.requestCodeFor(id)` (offset by 2000 to avoid collisions), letting alarms be scheduled/cancelled independently. `AlarmReceiver`/`BootReceiver` use `runBlocking` (not `goAsync()`) for Room work — small/bounded, and directly testable via plain `onReceive()`.
  - **Legacy migration**: `ui/playback/AlarmStateStore.kt` (SharedPreferences) is the pre-multi-alarm format. `AlarmRepository.migrateLegacyAlarmIfNeeded()` reads it once, inserts as a `WakeAlarm` row, clears the store — naturally idempotent. Called from `AlarmListViewModel.loadAlarms()`, `BootReceiver`, and `AlarmReceiver` (when a broadcast arrives with no `EXTRA_ALARM_ID`, since an app update doesn't clear `AlarmManager` registrations, only a reboot does).
  - `AlarmReceiver` fires, then reschedules the same alarm for tomorrow (this is what makes it daily). `BootReceiver` (`RECEIVE_BOOT_COMPLETED`) re-schedules every `enabled && streamUrl != null` row after reboot (alarms are cleared on reboot). `AlarmScheduler.nextTriggerMillis()` is `internal`, covered by `AlarmSchedulerTest`'s pure-function tests.
  - UI mirrors the station list/edit split: `AlarmListViewModel` + `AlarmListScreen` (Switch/delete with confirm dialog, FAB to add), `AlarmEditViewModel` + `AlarmEditScreen` (`AlarmEditActivity`'s `EXTRA_ALARM_ID`, `-1`/absent = new). `AlarmScheduler.schedule`/`cancel` are called from the Composable/Activity layer, not ViewModels (same "ViewModels don't hold a `Context`" rule as elsewhere). `save()` forces `enabled = false` if no station selected (silent, no error string). Delete button is available even if the alarm's station was removed from the library.
  - **Testing**: `AlarmReceiverTest`/`BootReceiverTest` hit the real `AppDatabase.getDatabase()` singleton (no injection seam in a `BroadcastReceiver`) — `@Before` defensively deletes all alarms to avoid cross-test leakage.
- Both `Switch`es (`SettingsScreen`'s metered toggle, `AlarmScreen`'s enable toggle) share one explicit `SwitchDefaults.colors(...)` — the Material3 default checked-track color (`md_theme_dark_primary`) is nearly identical to `glass_accent`, making a checked thumb nearly invisible against its track. Reuse this color set for any 3rd `Switch`.
- **Settings screen** (`ui/SettingsScreen.kt`'s `SettingsActivity`) hosts Export/Import stations (logic unchanged, just relocated from `MainScreen`/`MainViewModel`) and the one setting: `ui/playback/SettingsStore.kt` persists `warnOnMeteredConnection: Boolean` (default `true`). The check (`isMeteredConnection()`, `ConnectivityManager.isActiveNetworkMetered`) lives in `MainScreen` and gates only the direct play-button tap on a `StationItem` — deliberately **not** the mini player/full-screen bar, widget, Android Auto, or the wake-up alarm (already-in-progress switches or background starts with no good confirmation surface).

## Localization

- `res/values/strings.xml` is the source of truth (English); `res/values-<lang>/strings.xml` (ru, es, zh-rCN) must mirror the exact same `name` keys — Android silently falls back to the default string for a missing key rather than erroring. Format-argument placeholders (`%1$s`, `%1$d:%2$02d`, ...) must match verbatim in position/type (args are passed positionally). `app_name` stays untranslated everywhere.

## Database & migrations

- `RadioStation` has unique indices on `name`/`streamUrl` (schema v3) on top of app-level `isNameTaken`/`isUrlTaken` checks. `AppDatabase` has `exportSchema = true` (JSONs in `app/schemas/`) — **schema changes must ship a real `Migration` + schema snapshot**, never `fallbackToDestructiveMigrationFrom` (only allowed for pre-2.0 installs jumping from v1, no v1 snapshot exists). `MIGRATION_2_3` is the dedupe-then-constrain template: delete pre-existing `name`/`streamUrl` collisions (keep oldest row) before creating the unique indices.
- `AppDatabaseMigrationTest` verifies migrations against hand-built `LegacyVN...` entity/DAO/DB snapshots (not the real `RadioStation`/`AppDatabase`, which already carry later columns) — needs `kspTest 'androidx.room:room-compiler:2.8.4'`. Templates: `MIGRATION_3_4` (plain `ALTER TABLE ADD COLUMN`), `MIGRATION_2_3` (dedupe-then-constrain), `MIGRATION_7_8` (full create/copy/drop/rename rebuild, needed to *drop* a column since `DROP COLUMN` support depends on the bundled SQLite version). `MIGRATION_8_9` (new `wake_alarms` table) reuses the real `RadioStation`/`RadioStationDao` in its `LegacyV8Database` since `radio_stations`' shape is unchanged at v8.
- **List ordering**: `RadioStation.sortOrder` (`Int` ascending, schema v8) plus drag-to-reorder replaced the old favorites/pinning (`isFavorite DESC, id ASC`) — a free manual order and auto-pin-to-top can't coexist. `RadioStationDao.getAllStations()` orders `sortOrder ASC, id ASC` (the single source of truth — list, export, Android Auto browse tree, widget skip buttons all use it).
  - `insertStation()` auto-assigns `sortOrder = getMaxSortOrder() + 1` (appends), overwriting the passed-in value. `restoreStation()` (used by `undoDelete()`) is the one exception — inserts as-is, preserving `id`/`sortOrder`.
  - `MainViewModel.moveStation()` mutates the in-memory list only (per-frame drag feedback, no DB write); `persistStationOrder()` commits via `updateSortOrder(orderedIds)` (`@Transaction`) once the drag ends.
  - Drag gesture: `ui/components/DragDropState.kt` (hand-rolled `detectDragGesturesAfterLongPress`, since this Compose BOM has no built-in `LazyColumn` reorder support). Attached to the whole `LazyColumn`; long-press-then-still naturally disambiguates from `StationItem`'s per-item `detectHorizontalDragGestures` swipe-to-reveal (immediate directional drag). Disabled entirely while `searchQuery` is non-blank (reordering a filtered subset is ambiguous).
  - `StationItem` shows a static `Icons.Default.DragIndicator` (visual affordance only, not a separate touch target). `isDragging` (computed in `MainScreen`) drives the dragged item's elevated `Card`/border and `graphicsLayer { translationY }`/`zIndex(1f)`.
- `RadioStation.genre` (schema v5, nullable `TEXT`): Discover-added stations get it from the directory's `tags` field; manual adds get an optional free-text field. Matched by `MainViewModel.filteredStations`'s search; shown in `StationItem`'s subtitle in place of the raw URL when present.
- `RadioStation.isHls` (schema v6, `Boolean` default `false`): sourced from the directory's `hls` flag on Discover add; manual adds always get `false`. Makes `RadioPlaybackService.isHlsUrl()` more reliable than its `"m3u8" in url` heuristic — see Station discovery.
- `RadioStation.radioBrowserUuid` (schema v7, nullable `TEXT`): the directory's `stationuuid`, same "Discover sets it, manual gets null" pattern — lets `RadioPlaybackService` register a "click" with the directory on play.

## Backup: export / share / import

- Backup JSON shape (`{name, streamUrl, customIcon, genre, isHls, radioBrowserUuid}[]`, `data/StationBackupJson.kt`) deliberately excludes `sortOrder` (not meaningful across devices — imports append via normal `insertStation()`) and `isFavorite` (older backups with that key just get it ignored on import). Used both by `exportStationsToJson()` (all stations) and per-station sharing (single in-memory `RadioStation`). `importStationsFromJson()` reads the same shape, wired to `SettingsScreen`'s Import (SAF `OpenDocument`, MIME filter `*/*` — none of the four importable formats have a reliable SAF MIME type via `content://` URIs).
- `RadioStationRepository.importStations()` is `SettingsScreen`'s single import entry point: sniffs the picked file's content (not extension/MIME) and dispatches to `importStationsFromJson()` for a JSON array (`content.trimStart().startsWith("[")`), else `importStationsFromPlaylist()`. The latter delegates format detection to `data/PlaylistImport.kt`, which tries — in order — XML containing an `<outline>` tag (OPML, via `android.util.Xml`'s pull parser, so `PlaylistImportTest` needs `RobolectricTestRunner` even with no `Context`, same reason as `org.json`), `#EXTM3U`-prefixed text (M3U/M3U8, `#EXTINF:duration,Title` line names the URL line that follows; a bare URL line uses itself as the name), or `[playlist]`-prefixed `key=value` text (PLS, `FileN`/`TitleN` pairs keyed by their numeric suffix, order-independent). Throws `IllegalArgumentException` if none match. Only `name`/`streamUrl` exist in any of these three formats, so every other `RadioStation` field stays at its entity default — same skip-on-duplicate/failed-on-missing-field semantics as JSON import.
- "Export stations" and per-station "Share" (swipe action) both go through `util/StationShare.kt` (`Intent.ACTION_SEND` file attachment, so it reaches Telegram/WhatsApp/email directly). Needs a `FileProvider` (`res/xml/file_paths.xml`, authority `com.urlradiodroid.fileprovider` hardcoded in `StationShare` — keep in sync with `applicationId`). `testOptions.unitTests.includeAndroidResources = true` is required for Robolectric to see it. `FileProvider` statically caches resolved roots per authority for the process lifetime, so `StationShareTest` uses a single test method rather than one per `@Test`.
- Track title (ICY metadata via `StreamRecorder`, shown in `PlaybackScreen`) has a copy-to-clipboard button (`LocalClipboardManager`). `PlaybackScreen` and `NowPlayingBottomBar`'s `MiniPlayerCardFull` both use `Modifier.basicMarquee(maxLines = 1)` for overflow. The notification's own title/text has **no marquee equivalent** — Android disables `RemoteViews` text animation for 3rd-party apps — but Android 11+'s system media-controls panel (drawn by SystemUI from `MediaSession` metadata, not the `Notification` object) does auto-scroll, which is why `updateMediaItemMetadataForTrack()` keeps `MediaMetadata.setArtist(trackTitle)` current on every ICY change.

## Station discovery & stream validation

- `data/RadioBrowserApi.kt`: minimal client for the [Radio Browser API](https://api.radio-browser.info/) (own `OkHttpClient`, `org.json`, not Retrofit/DI). Hits `https://all.api.radio-browser.info/` (round-robins mirrors). `baseUrl`/`client` are constructor params for `MockWebServer` tests.
- `ui/DiscoverStationsViewModel` + `ui/DiscoverStationsScreen.kt` (`DiscoverStationsActivity`, launched from a standalone `AssistChip` atop `MainScreen`'s list): search by name/genre/country/language/nearby, add straight into the library. Chip order `NAME, NEARBY, GENRE, COUNTRY, LANGUAGE`. Debounced 400ms (`SEARCH_DEBOUNCE_MS`). Doesn't `finish()` after each add (multi-add browsing session) — `MainActivity`'s `discoverStationsLauncher` reloads unconditionally on return. Name collisions on add auto-disambiguate (`"Station (2)"`, ...).
- `util/StreamValidator.kt` probes a URL before `AddStationViewModel.save()` commits (HEAD, falling back to GET for mounts that reject HEAD with 405), closing the response as soon as headers arrive. Runs after the cheap name/URL-taken DB checks; surfaces `error_stream_unreachable` on failure.
- `RadioBrowserStation.hls`/`isHls`: `onStartCommand`/`startPlayback`/`applyPlayback` only ever receive a bare stream URL (every caller passes `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` strings, never a full `RadioStation`), so `onStartCommand` resolves the HLS hint via `repository.getStationByUrl()` before calling `startPlayback`. Android Auto's `playFromBrowseTree`/`onAddMediaItems` pass `station.isHls` straight from in-memory `cachedStations`, skipping the DB round-trip. A network-loss retry reuses the cached `currentKnownHls` rather than re-resolving (so retry can't flip-flop the decision).
- `RadioBrowserApi.registerClick(uuid)` reports a play to the directory's `clickcount` metric (separate from `votes`, which `search()` sorts by). Fire-and-forget, swallows `IOException`. `radioBrowserUuid` is resolved via the same `getStationByUrl()`/`cachedStations` plumbing as `isHls`. Only fired from direct-play (`onStartCommand`) and `playFromBrowseTree` — **not** from process-restart resume (`playbackStateStore.restore()`), to avoid double-counting on cold start.
- `util/CountryFlagEmoji.from(code)`: pure function, 2-letter ISO code → flag emoji via the Unicode regional-indicator offset, `null` for anything malformed. `DiscoverStationsScreen`'s `stationSubtitle()` shows just the flag (no country name text) when available; `codec` folds into the same subtitle's bitrate segment (`"128 kbps MP3"`).
- `RadioBrowserStation.votes` feeds a star badge (`util/VoteCountFormatter.format()`) shown when `votes > 0`; `search()` sorts by `order=votes` so the visible ranking matches the badge.
- `RadioBrowserStation.homepage` surfaces as an `OpenInNew` icon button (`Intent.ACTION_VIEW`, caught against `ActivityNotFoundException`). Presentation-only, not persisted.
- `RadioBrowserStation.favicon` feeds the icon pipeline for Discover adds: `RadioBrowserApi.downloadFavicon()` (returns `null` on any failure, never throws) + `IconStorage.saveImageBytes()` (shares `saveImage()`'s downscale-to-512px-JPEG internals). Fired fire-and-forget from `addStation()` (child of the insert's `viewModelScope.launch`) — station shows the auto-generated emoji immediately, `customIcon` backfills once the download finishes or never if it fails. `DiscoverStationsViewModel` is the only ViewModel holding a `Context` (`appContext`, always `applicationContext`), needed for `IconStorage`.
- **"Near me" search** (`DiscoverSearchMode.NEARBY`): no text query — `DiscoverStationsScreen` hides the `TextField` and drives a one-shot permission-then-search flow. A `LifecycleEventObserver` retries `requestNearbySearch()` on every `ON_RESUME` while `mode == NEARBY` and the last failure was `discover_location_unavailable` (so returning from Settings with GPS now on "just works"); reads `uiState` via `rememberUpdatedState` since the `DisposableEffect` is keyed only on `lifecycleOwner`. `RadioBrowserApi.searchNearby()` uses `geo_lat`/`geo_long`/`geo_distance`/`order=distance`; `distanceKm` populates only for this call, shown as `"📍 2.3 km"`. Search radius (`NEARBY_RADIUS_METERS`) is hardcoded 50km.
- `util/LocationProvider.kt` wraps plain `android.location.LocationManager` (not `FusedLocationProviderClient` — avoids a new Play Services dependency). Prefers cached `getLastKnownLocation()` over `requestSingleUpdate()`; never requests continuous updates or persists a coordinate. Gates on `ACCESS_COARSE_LOCATION` only (deliberate privacy tradeoff — city-level precision is enough). `DEFAULT_TIMEOUT_MS = 20s` (a cold-start GPS fix can realistically take 15-20s; 10s cut searches off prematurely). **Emulator note**: the simulated GPS provider produces no fix from the in-app location toggle alone — set a location via Extended Controls → Location.
- Permission flow lives in `DiscoverStationsScreen` (ViewModel has no Android permission APIs): `ContextCompat.checkSelfPermission` → rationale `AlertDialog` if ungranted → `RequestPermission()` launcher. On grant, calls `searchNearby()` directly; on denial, `onLocationPermissionDenied()` sets an inline hint under the chip row.
- `LocationProviderTest` needs two non-obvious Robolectric fixes: (1) permission grant/deny via `Shadows.shadowOf(application).grantPermissions()/denyPermissions()` — `ShadowApplicationPackageManager.grantRuntimePermission()` doesn't affect `ContextCompat.checkSelfPermission()` under this Robolectric version; (2) `ShadowLocationManager.simulateLocation()` posts to the `Looper` it was registered against, and Robolectric's main looper is paused by default — needs an explicit `Shadows.shadowOf(Looper.getMainLooper()).idle()` after `simulateLocation()`, or the callback never fires (silently eaten by the internal 5s timeout).
- `RadioBrowserStation.sslError` surfaces as a `Warning` icon (tinted `colorScheme.error`) next to the homepage link — non-blocking, since the directory's HTTPS check failing doesn't guarantee ExoPlayer will also fail. Presentation-only.

## Android Auto

- `RadioPlaybackService extends MediaLibraryService` (not plain `MediaSessionService`) so the station list is browsable from a head unit. Manifest intent-filter action is `androidx.media3.session.MediaLibraryService` (plus `android.media.browse.MediaBrowserService` for legacy clients) — `MediaLibraryService.SERVICE_INTERFACE` is a genuinely different string, not a supertype the old filter would still match. Service is `android:exported="true"` (Auto binds cross-process); `res/xml/automotive_app_desc.xml` + a `com.google.android.gms.car.application` `<meta-data>` makes the app eligible to show up in Auto.
- `onBind()`: `MediaLibraryService.onBind()` returns the real session binder for session-interface intents (Auto, Assistant, legacy browsers) and `null` for our own Activities' plain explicit-component bind. Must be `return super.onBind(intent) ?: binder` — unconditionally returning our own `LocalBinder` breaks Auto/Assistant's connection.
- A `MediaLibrarySession` must exist immediately in `onCreate()` (`buildMediaSession(streamUrl = null)`) — `onGetSession()` must return non-null for Auto to browse even before anything has played. `buildMediaSession()` still releases/rebuilds on every real `startPlayback()`; the `null` case skips station extras on the session's `sessionActivity` intent, relying on `PlaybackActivity` reading the live station once bound.
- Browse tree is flat (root → all stations, `sortOrder ASC, id ASC`) via `onGetLibraryRoot`/`onGetChildren`/`onGetItem`. Station media items use `mediaId = streamUrl`.
- Tapping a station reaches `onAddMediaItems` with a bare `MediaItem`. This is **not** routed through `startPlayback()`/`buildMediaSession()` — rebuilding the session whose callback is currently executing isn't supported. Instead `startPlayback()` is split into `buildMediaSession(streamUrl)` (top-level entry points only: `onCreate`, `onStartCommand`) + `applyPlayback(streamUrl, isRetry)` (player/notification/timeshift/retry pipeline, safe to call reentrantly). `onAddMediaItems` resolves the tapped `mediaId` against `cachedStations`, calls `applyPlayback()` directly, and returns an empty item list.
- `loadBrowsableStations()`/`playFromBrowseTree()` are `internal` (not `private`) so `RadioPlaybackServiceAutoTest` can exercise the DB → media item → playback pipeline directly — media3's own `MediaBrowser` Binder round-trip tests need instrumentation (`androidTest`), which this project doesn't have.

## Home screen widget

- `widget/RadioWidget.kt`: Jetpack Glance `GlanceAppWidget` (`widget/RadioWidgetReceiver.kt` + `res/xml/radio_widget_info.xml`; `android:initialLayout` is a required-but-unused placeholder, Glance replaces it at render time). Shows last-played/current station + play-pause + skip, dark palette matching `ui/theme/Color.kt`.
- `ui/playback/WidgetStateStore.kt` (separate from `PlaybackStateStore` on purpose): `PlaybackStateStore`'s contract is "last *started*, cleared on explicit stop" (for process-death resume); the widget wants to keep showing the last station with a play affordance even after an explicit stop. `RadioPlaybackService.updateWidget()` writes it and calls `RadioWidget().updateAll()` at the end of every `refreshSnapshot()`.
- Button taps are `ActionCallback`s in `widget/WidgetActions.kt`: `TogglePlaybackAction` (stops via `RadioPlaybackService.ACTION_STOP` — a widget `ActionCallback` isn't a bound instance, so it can't call `stopPlayback()` directly) or resumes; `NextStationAction`/`PreviousStationAction` use `util/StationNavigator.kt` (pure, JUnit-testable next/prev index calc), wrapping at either end. `ActionCallback` requires a no-arg constructor (same as `Service`/`BroadcastReceiver`), so each resolves `RadioStationRepository.create(context)` itself.
- Testing: `RadioWidgetTest` uses `runGlanceAppWidgetUnitTest` against `RadioWidgetContent` (`internal`, node-tree assertions, no rendering). `WidgetActionsTest` calls `ActionCallback.onAction()` directly, asserting via `shadowOf(application).nextStartedService`. Both this and `RadioPlaybackServiceAutoTest`'s next/prev test hit the real `AppDatabase.getDatabase()` singleton (no injection seam) — each sticks to a single `@Test` method and defensively clears rows before seeding.

## Testing

- Unit tests only (`app/src/test`, no `androidTest`); CI's `test` job runs `./gradlew test`. Room DB tests and anything needing a `Context` use Robolectric (`@Config(sdk = [29])`).
- `StreamRecorder`/`TimeshiftController`/`StreamValidator`/`RadioBrowserApi` tests use a local `MockWebServer` (all four construct their own `OkHttpClient`, not injected). Any test touching `org.json` needs `RobolectricTestRunner` even without a `Context` — the plain Android stub jar throws `RuntimeException: not mocked` on real `org.json` calls.
- `RadioPlaybackService`'s pure decision logic (`isHlsUrl`, `retryDelayMs`, `isRetryableNetworkError`, `loadBrowsableStations`, `playFromBrowseTree`) is `internal` so tests can call it without the full `MediaLibraryService`/ExoPlayer lifecycle — though `RadioPlaybackServiceAutoTest` does construct the full service via `Robolectric.buildService(...).create().get()` (real `ExoPlayer`+`MediaLibrarySession` come up fine under Robolectric; it's specifically the `MediaBrowser` Binder round-trip that's avoided).
- ViewModel tests exercising a suspend fn that hops onto real `Dispatchers.IO` (`StreamValidator`/`RadioBrowserApi` calls) must poll for expected state (`awaitTrue { condition }` wrapped in `withTimeout`) rather than a single `advanceUntilIdle()`, which only fast-forwards the test dispatcher's own queue.
- Even fully-virtual tests (`StandardTestDispatcher` + in-memory Room) need a *content*-based poll condition, not count-based, when asserting after a `viewModelScope.launch` update that doesn't change collection size (e.g. a `sortOrder` reorder) — a count-only condition can already be true before the coroutine runs, returning without ever driving the scheduler. Poll the actual field expected to change.
- `IconStorageTest` opts into `@GraphicsMode(GraphicsMode.Mode.NATIVE)` (like screenshot tests) since plain Robolectric fakes `BitmapFactory` decode results — needed to assert real downsampled pixel dimensions.
- **Never call a suspend function inside `awaitTrue`'s poll lambda** (or any tight `while (!cond) { advanceUntilIdle() }` loop) — the poll itself would suspend on a real background thread, and nothing drives the virtual scheduler forward while you're suspended: a real deadlock, only broken by `withTimeout`. When there's no synchronous side effect to poll, poll a plain `StateFlow` field twice in sequence (transient "in progress" value, then it flipping back) rather than reading the DB from inside the poll.
- `advanceUntilIdle()`-busy-looping stops being reliable once a background chain does more than one real-dispatcher hop back-to-back (network IO → `withContext(IO)` → sequential Room calls) — repeated calls with no elapsed wall-clock time can race ahead of a still-in-flight later hop. `delay()`-based retries don't fix this either (`delay`/`withTimeout` under `runTest` use virtual time, never truly sleep). Fix: a plain `while` loop bounded by real `System.currentTimeMillis()`, reading state via `runBlocking { ... }` (a genuinely separate blocking call, safe here since it doesn't participate in the virtual scheduler) with a real `Thread.sleep(20)` between attempts.
- Screenshot tests (`StationItemScreenshotTest`) use Roborazzi over Robolectric Native Graphics (`@GraphicsMode(GraphicsMode.Mode.NATIVE)`). Capture with the plain `captureRoboImage { content }` lambda from `roborazzi-compose`, not `createComposeRule()`/`createAndroidComposeRule<ComponentActivity>()` — this manifest has no `ComponentActivity` launcher entry, and the lambda form self-registers `RoborazziActivity` at runtime instead. Reference images live in `app/src/test/screenshots/` (`build/` is gitignored, so the Roborazzi default output dir wouldn't be). `./gradlew recordRoborazziDebug` re-records after an intentional visual change (review the diff); `./gradlew verifyRoborazziDebug` (in CI's `test` job) fails on any pixel diff — plain `./gradlew test` doesn't. **Baselines are recorded on Ubuntu** — font-rendering differs enough by platform that running `verifyRoborazziDebug` locally on macOS fails with a full-image diff even with zero relevant changes; don't re-record from a local non-Ubuntu run.
- **Before pushing any change that touches a Composable covered by a screenshot test** (currently `StationItem`, via `StationItemScreenshotTest`): `./gradlew test` alone is not sufficient sign-off — it runs Roborazzi's capture step but not `verifyRoborazziDebug`, so a changed icon/border/color silently passes locally and only fails in CI. Since baselines can't be correctly re-recorded on macOS (see above), the fix-forward loop is: push, let CI's `verifyRoborazziDebug` fail and name the affected tests, then re-add the temporary `record-screenshots.yml` workflow (`workflow_dispatch` running `./gradlew recordRoborazziDebug` on `ubuntu-latest`, see git history: `27d145f`, `03cd491`), trigger it, download the `roborazzi-screenshots` artifact, replace the PNGs in `app/src/test/screenshots/`, commit, and delete the temporary workflow again.
