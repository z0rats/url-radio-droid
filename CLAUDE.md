# CLAUDE.md

Android app (Kotlin, Jetpack Compose) for playing internet radio streams (HTTP/HTTPS, incl. HLS `.m3u8`) via ExoPlayer/Media3, with local station storage in Room and background playback.

## Project layout

- Single module: `app/`. Dependency versions live in `gradle/libs.versions.toml` (Gradle auto-wires the `libs` accessor from this conventional path — no extra `settings.gradle` config needed); reference as `libs.foo.bar` / `alias(libs.plugins.foo)`, don't hardcode a `group:artifact:version` string in `build.gradle`/`app/build.gradle` again. Compose artifacts without a `version.ref` in the catalog (`androidx-compose-ui`, `-material3`, etc.) are intentionally unversioned there — they're constrained by the `androidx.compose:compose-bom` platform instead, so bumping Compose is a one-line `composeBom` version change, not touching each artifact.
- `app/src/main/java/com/urlradiodroid/`: `data/` (Room + repository), `ui/` (Compose screens + `ui/components`, `ui/playback`, `ui/theme`), `util/`.
- Lint: ktlint (`ktlintCheck` / `ktlintFormat`); `.editorconfig` allows PascalCase-looking names on `@Composable` functions (`ktlint_function_naming_ignore_when_annotated_with = Composable`). CI: `.github/workflows/ci.yml` (test + Roborazzi screenshot verify, lint, `assembleDebug` on every push/PR), `release.yml` (tag `v*` → signed `assembleRelease`, attached to a GitHub Release).
- `gradlew`/`gradlew.bat`/`gradle-wrapper.jar` are committed — use `./gradlew` locally to match CI's Gradle version.
- `app/src/main/baselineProfiles/baseline-prof.txt` + the `androidx.profileinstaller` runtime dependency give a Baseline Profile for cold start: at install time `profileinstaller` hands ART the human-readable rules in that file so it AOT-compiles the listed classes/methods instead of interpreting/JITting them on first launch. AGP 9.x reads this from the `baselineProfiles/` source-set *directory* (any number of `.txt` files merged together), not the older single `src/main/baseline-prof.txt` path — putting it there silently produces an empty profile with no error, so if `mergeReleaseArtProfile`'s output stops containing `com/urlradiodroid` entries, check the file didn't drift back to the old path. It's hand-curated (two wildcard rules covering all of `com.urlradiodroid.**`, keeping it bounded to our own code rather than guessing exact Compose/Room/Media3 internals — AGP's wildcard expansion resolves these against the real compiled classes at build time, verify with `./gradlew :app:expandReleaseArtProfileWildcards` and grep the output for `com/urlradiodroid`) rather than generated by a `androidx.benchmark.macro.junit4.BaselineProfileRule` — that needs a separate instrumented (`androidTest`) benchmark module running on a real device/emulator, which this project deliberately doesn't have (see Testing below). Re-curate by hand if `com.urlradiodroid`'s package structure changes shape significantly; there's no automated regeneration step.
- Release builds (`app/build.gradle` `buildTypes.release`) have `minifyEnabled true` + `shrinkResources true` (R8, relying on consumer proguard rules bundled in the Room/Media3/Compose AARs — `app/proguard-rules.pro` is still the stock empty template, nothing project-specific has been needed so far) and are signed via `signingConfigs.release`, which reads `RELEASE_KEYSTORE_PATH`/`RELEASE_KEYSTORE_PASSWORD`/`RELEASE_KEY_ALIAS`/`RELEASE_KEY_PASSWORD` from the environment — if `RELEASE_KEYSTORE_PATH` isn't set (e.g. a local `assembleRelease`), the signingConfig is skipped entirely and Gradle produces an unsigned `app-release-unsigned.apk` rather than falling back to debug-signing. `release.yml` decodes the keystore from the `RELEASE_KEYSTORE_BASE64` GitHub Secret to a workspace-local file (deleted with `if: always()` after the build) and supplies the other three as env vars from their own secrets — the actual `keystore/release.keystore` is gitignored and never committed; losing it means future releases can't be signed with the same key, so it (and its password) must be kept in a password manager, not just on the machine that generated it.

## Architecture conventions

- `data/RadioStationRepository` is the only class that talks to `RadioStationDao`/`AppDatabase` directly — `ViewModel`s take a `RadioStationRepository`, never an `AppDatabase`/`Context`. Created via `RadioStationRepository.create(context)`.
- Two `ViewModel` styles are in use, pick based on complexity: `MainViewModel` exposes plain `StateFlow`s and fire-and-forget `fun`s (simple CRUD/search state, no validation) plus, where a screen needs one-shot feedback, a `Channel`-backed `events: Flow<MainScreenEvent>` alongside them rather than switching the whole ViewModel to a `UiState` data class. `AddStationViewModel` uses the fuller `UiState` data class + `Channel`-backed `events: Flow<AddStationEvent>` (`SaveSucceeded`/`SaveFailed`) for one-shot UI effects (validation errors, toasts, navigating back on success) — follow whichever shape matches the screen's complexity: a bare event channel bolted onto plain `StateFlow`s for simple CRUD screens, the full `UiState`-plus-events shape for screens with form validation.
- Deleting a station is immediate but undoable, not gated behind a confirmation dialog: `MainViewModel.deleteStation(station)` deletes from the DB right away, reloads the list, then emits `MainScreenEvent.StationDeleted(station)`; `MainScreen` collects `viewModel.events` and shows a `Snackbar` ("Station deleted" / "Undo") via `SnackbarHostState` — if the user taps Undo before the snackbar times out, `MainViewModel.undoDelete(station)` re-inserts the same `RadioStation` object (including its original `id`), which SQLite accepts as an explicit rowid for an `autoGenerate` primary key as long as nothing reused it since, so the restored station reappears in its original position (`getAllStations()` orders by `id ASC` within the same-favorite group) rather than jumping to the end of the list.
- `RadioPlaybackService` (the `MediaSessionService`) exposes a `playbackSnapshot: StateFlow<PlaybackSnapshot>` (isPlaying, isBuffering, currentMediaId, hasTimeshift, isAtLive, trackTitle, sleepTimerEndAtMs) as the single source of truth for reactive UI state — screens collect this instead of polling the service. It delegates to two extracted collaborators: `playback/TimeshiftController` (buffer file lifecycle + seek math for the rewind/live feature) and `playback/PlaybackStateStore` (SharedPreferences-backed; remembers the last-started station so `onStartCommand` can resume playback if the system kills and restarts the service with a null intent — this is what makes background playback survive process death, not just app backgrounding).
- Any `Activity` that binds `RadioPlaybackService` and passes the instance into a `@Composable` (`MainActivity`, `PlaybackActivity`) must hold it as `mutableStateOf<RadioPlaybackService?>`, not a plain `var`. `ServiceConnection.onServiceConnected` fires asynchronously after the first composition; a plain field mutation there is invisible to Compose, so the composable (and anything derived from it via `playbackSnapshot`, e.g. `isPlaying`/track title/sleep timer) stays stuck at its initial `null` forever.
- Reconnect-on-network-loss uses capped exponential backoff (`RadioPlaybackService.retryDelayMs`: 2s/4s/8s/16s, capped at 30s, `MAX_RETRY_COUNT = 5`) rather than retrying forever — after the limit it surfaces "Connection failed" and stops instead of hammering the stream.
- Station icons (`RadioStation.customIcon`, doc comment: "Emoji string or image file path") default to auto-generated from the name/URL via `util/EmojiGenerator` (keyword + hash fallback), but can be overridden via the picker dialog (`ui/components/StationIconPickerDialog.kt`, opened by tapping the circular preview at the top of `AddStationScreen`) — "Choose Emoji" picks from `EmojiGenerator.pickerEmojis`, "Upload Image" launches the system photo picker (`ActivityResultContracts.PickVisualMedia`, no runtime permission needed) and `util/IconStorage.saveImage()` downscales the picked image to at most 512px and copies it as a JPEG into `filesDir/station_icons/`, "Remove Icon" clears it back to the auto-generated fallback. A leading `/` is what `IconStorage.isImagePath()` uses to tell a stored file path apart from an emoji string (no emoji starts with that character), and every render site (`StationItem`, `NowPlayingBottomBar`'s two mini-player variants, `AddStationScreen`'s preview — via the shared `rememberStationIconBitmap()` composable in `ui/components/StationIcon.kt`) falls back to the emoji path if the file is missing/corrupt. `AddStationViewModel` only deletes the *replaced* icon file, and only once `save()` actually commits (compares the freshly-loaded `originalCustomIcon` against the session's final value) — it deliberately does not clean up on cancel/back or on intermediate re-picks before saving, so an abandoned edit can leak a small orphaned JPEG; accepted as a minor, bounded cost rather than adding cleanup-on-cancel plumbing. **Not yet wired up:** `PlaybackScreen` (full-screen player) and `RadioPlaybackService`'s notification large-icon still only show the auto-generated emoji — neither has ever taken `customIcon` into account (this predates the picker), and threading a custom image through the notification/media-session pipeline (`EXTRA_STATION_ICON` on every intent that starts/resumes playback, `PlaybackStateStore` persistence, decoding a bitmap in `getCurrentLargeIcon`) is a larger, separate change.
- `ui/theme/Dimens.kt` holds the design tokens: `Spacing` (4/8/16/24dp scale — only use where a padding/gap value already matches one of these exactly, don't force-fit one-off values like a 20dp chip padding into the scale) and `AppShapes` (a Material3 `Shapes` wired into `MaterialTheme(shapes = AppShapes)` in `Theme.kt`: `extraSmall`=8dp for icon-sized controls, `medium`=16dp for buttons/fields/chips, `large`=24dp for primary cards, `extraLarge`=40dp for the pill-shaped mini-player cards). Reference them as `MaterialTheme.shapes.large` / `Spacing.md` instead of new `RoundedCornerShape(Ndp)`/raw `.dp` literals so radii and rhythm stay consistent across screens.
- `URLRadioDroidTheme` is always-dark by design (no system-theme following, no light `ColorScheme`, no `dynamicColor` param) — `card_surface`/`text_primary`/etc. in `Color.kt` are hardcoded dark values that assume this, so a default Material component left without explicit `colors` (a `DropdownMenu`, a `TextField`) would go invisible-on-light if the theme ever did follow the system. Don't reintroduce `isSystemInDarkTheme()` or a light `ColorScheme`; if a screen needs to look different, do it with an explicit param, not by resurrecting system-theme following.
- When a `Modifier` chain combines a rounded `shape` with `.clickable()`/ripple (or is passed as a `Card`'s `modifier`), the shape-clip must come before the click handler in the chain — otherwise the ripple/press-state paints the full rectangular bounds and bleeds past the rounded corners. Prefer `Card(onClick = ..., border = BorderStroke(...))` (shape-aware internally) over chaining `Modifier.border()`/`.clickable()` onto a `Card`'s incoming `modifier`; for plain `Box`/`Row` click targets, order it `.clip(shape).clickable(...)`.
- `util/AppShortcuts.kt` publishes up to 2 dynamic App Shortcuts (long-press the launcher icon): the last-played station (from `playback/PlaybackStateStore`) and the top favorite (`stations.firstOrNull { it.isFavorite }`, same ordering as the DB query), deduped to one if they're the same station. `AppShortcuts.refresh(context, stations)` is called from a `LaunchedEffect(allStations)` in `MainScreen` — it re-fires on every `loadStations()` (including plain `onResume()`), which is also the only point the last-played shortcut picks up a station started elsewhere in the same session; there's no dedicated observer on `PlaybackStateStore`, so a shortcut can lag until the user next opens `MainActivity`, same "good enough" freshness bar as this codebase's other minor accepted gaps (e.g. the icon-picker's orphaned-file cleanup above). Both shortcuts always show the auto-generated emoji, never `RadioStation.customIcon` — the last-played entry only has a name/URL from `PlaybackStateStore`, not a full `RadioStation`, so there's nothing to read a custom icon off of; matches the existing scope boundary where the notification/`PlaybackScreen` also don't show `customIcon`. Each shortcut's intent targets `PlaybackActivity` with `EXTRA_AUTO_PLAY = true`, which makes `handleIntent()` call `togglePlayback()` immediately instead of just opening the screen and waiting for a tap — **this requires `PlaybackActivity` to be `android:exported="true"`** (a hard OS requirement: `ShortcutManagerCompat.setDynamicShortcuts` throws if a shortcut's target activity isn't exported), so any other installed app can now also launch it with arbitrary `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` extras and make this app start playing an arbitrary stream URL — accepted as low-severity (no data access, no permission escalation, same trade-off most media-player "quick play" shortcuts/widgets make) rather than adding a signature-permission gate.

## Database & migrations

- `RadioStation` has unique indices on `name` and `streamUrl` (DB-level, added in schema v3) on top of the pre-existing app-level `isNameTaken`/`isUrlTaken` checks — defense in depth against any insert path that bypasses the UI checks. `AppDatabase` is now on `exportSchema = true` (schema JSONs checked into `app/schemas/`, `room.schemaLocation` set via the `ksp {}` block in `app/build.gradle`) — **schema changes must ship a real `Migration` and a matching schema snapshot, not rely on `fallbackToDestructiveMigrationFrom`** (only allowed for pre-2.0 installs jumping from version 1, since there's no v1 schema snapshot to migrate from safely). `AppDatabase.MIGRATION_2_3` is the template to follow: dedupes any pre-existing `name`/`streamUrl` collisions (`DELETE ... WHERE id NOT IN (SELECT MIN(id) ... GROUP BY ...)`, keeping the oldest row) before creating the unique indices, so upgrading never crashes or silently drops the wrong rows even if duplicates somehow predate the constraint.
- `AppDatabaseMigrationTest` verifies migrations against a hand-built copy of the pre-migration schema (kept in sync with `app/schemas/.../2.json`, `.../3.json`) rather than `MigrationTestHelper`, since that needs an `androidTest` source set with schema assets + instrumentation, which this project deliberately doesn't have (unit-tests-only, see Testing below) — needs `kspTest 'androidx.room:room-compiler:2.8.4'` so KSP generates Room impl classes for the test-only entities/DB it defines. Each migration gets its own `LegacyVN...` entity/DAO/DB snapshot in that file (not the real `RadioStation`/`AppDatabase`, which already carry later columns/indices and would make the "legacy" database already match the target schema, defeating the point of the test) — `AppDatabase.MIGRATION_3_4` (adds `isFavorite`) is the template for a plain `ALTER TABLE ... ADD COLUMN` migration, versus `MIGRATION_2_3`'s dedupe-then-constrain pattern.
- `RadioStation.isFavorite` (schema v4) drives the favorites/pinning feature: `RadioStationDao.getAllStations()` orders `isFavorite DESC, id ASC` so pinned stations float to the top while preserving insertion order within each group — this is the single source of truth for list ordering, not a client-side sort, so it applies everywhere `getAllStations()` is used (list, export). Toggled via `MainViewModel.toggleFavorite()` → `RadioStationDao.setFavorite()`, surfaced as a star icon on `StationItem` (next to the play button, not behind the edit/share/delete swipe reveal, since favorite status needs to be visible at a glance and togglable in one tap). Included in the backup JSON shape (`StationBackupJson`) and read back via `optBoolean("isFavorite", false)` so older backups without the field still import cleanly.

## Backup: export / share / import

- Backup JSON shape (`{name, streamUrl, customIcon}[]`) is centralized in `data/StationBackupJson.kt`, used both by `RadioStationRepository.exportStationsToJson()` (all stations, from DB) and by per-station sharing (serializes a single in-memory `RadioStation`, no DB round-trip). `RadioStationRepository.importStationsFromJson()` reads the same shape (JSON via `org.json`) and is wired to MainScreen's overflow menu "Import stations" (SAF `OpenDocument`).
- "Export stations" (overflow menu, all stations) and per-station "Share" (swipe action on `StationItem`, one station) both go through `util/StationShare.kt`, which sends the JSON as a file attachment via `Intent.ACTION_SEND` (so it can go straight to Telegram/WhatsApp/email/etc., not just to a chosen SAF location) — the file name is the station name for per-station shares. Needs a `FileProvider` (declared in `AndroidManifest.xml`, paths in `res/xml/file_paths.xml`, authority hardcoded as `com.urlradiodroid.fileprovider` in `StationShare` — must stay in sync with `applicationId` since there's only one build variant). `testOptions.unitTests.includeAndroidResources = true` in `app/build.gradle` is required for Robolectric to see manifest-declared components (providers, etc.) in tests — without it `FileProvider` can't find its meta-data at all. Note: `FileProvider` statically caches its resolved roots per authority for the process lifetime, so `StationShareTest` deliberately uses a single test method rather than one Activity/cacheDir per `@Test`.
- Currently playing track title (shown in `PlaybackScreen`, sourced from ICY in-band metadata parsed by `StreamRecorder`) has a copy-to-clipboard button next to it, using Compose's `LocalClipboardManager` — no `ClipboardManager` system service needed.

## Station discovery & stream validation

- `data/RadioBrowserApi.kt` is a minimal client for the [Radio Browser API](https://api.radio-browser.info/) directory (constructs its own `OkHttpClient`, parses with `org.json`, not injected as a `Retrofit`/DI dependency — same low-ceremony style as `StationBackupJson`). It hits `https://all.api.radio-browser.info/` by default, which round-robins across every community mirror, so no manual DNS-SRV server discovery is needed. `baseUrl`/`client` are constructor params purely so tests can point it at a `MockWebServer`.
- `ui/DiscoverStationsViewModel` + `ui/DiscoverStationsScreen.kt` (`DiscoverStationsActivity`, launched from a standalone `AssistChip` at the top of MainScreen's list, not the overflow (⋮) menu — it's a headline feature and that row otherwise had dead space above the search field) let the user search the directory by name/genre/country and add results straight into the library. Typing is debounced 400ms before calling the API (`SEARCH_DEBOUNCE_MS`). Unlike `AddStationActivity`, this screen doesn't `finish()`/return a result after each add — the user browses and adds multiple stations in one visit, so `MainActivity`'s `discoverStationsLauncher` reloads the station list unconditionally on return (no `RESULT_OK` gate) rather than following `addStationLauncher`'s pattern. Name collisions on add are auto-disambiguated (`"Station (2)"`, `"Station (3)"`, ...) rather than blocking the add or silently overwriting.
- `util/StreamValidator.kt` probes a URL before `AddStationViewModel.save()` commits it (HEAD first, falling back to GET for Icecast/Shoutcast mounts that reject HEAD with 405) — it closes the response as soon as headers arrive without reading the effectively-infinite stream body. This runs after the existing name/URL-taken checks (cheap DB lookups first, network probe only if those pass) and surfaces `error_stream_unreachable` on failure instead of letting a bad URL surface as "Connection failed" the first time the user hits play.

## Testing

- Unit tests only (`app/src/test`, no `androidTest`); CI's `test` job runs `./gradlew test`. Room DB tests and anything needing a `Context` use Robolectric (`@RunWith(RobolectricTestRunner::class)`, `@Config(sdk = [29])`).
- `StreamRecorder`/`TimeshiftController`/`StreamValidator`/`RadioBrowserApi` tests spin up a local `okhttp3.mockwebserver.MockWebServer` instead of mocking OkHttp, since all four classes construct their own `OkHttpClient` internally (not injected). Any test that touches `org.json` (e.g. `RadioBrowserApiTest`) needs `@RunWith(RobolectricTestRunner::class)` even if it doesn't otherwise need a `Context` — the plain Android SDK stub jar used for non-Robolectric unit tests throws `RuntimeException: ... not mocked` on real `org.json` calls.
- `RadioPlaybackService`'s pure decision logic (`isHlsUrl`, `retryDelayMs`, `isRetryableNetworkError`) is `internal` (not `private`) specifically so it's callable from tests without exercising the full `MediaSessionService`/ExoPlayer lifecycle.
- ViewModel tests that exercise a suspend function hopping onto the real `Dispatchers.IO` (anything using `StreamValidator`/`RadioBrowserApi`, whose network calls run off the virtual `TestCoroutineScheduler`) must poll for the expected state (`awaitTrue { condition }` wrapped in `withTimeout`, see `DiscoverStationsViewModelTest`) rather than a single `advanceUntilIdle()`, which only fast-forwards the test dispatcher's own queue and won't wait for real background-thread work to finish.
- Even fully-virtual `MainViewModelTest`-style tests (`StandardTestDispatcher` + in-memory Room, no real dispatcher crossing) need a *content*-based poll condition, not a count-based one, when asserting after a `viewModelScope.launch`-fired update that doesn't change the collection size (e.g. `toggleFavorite`) — `waitForStationsCount(viewModel, sameSizeAsBefore)`'s while-loop condition is already true before the launched coroutine runs, so it returns without ever calling `advanceUntilIdle()`, and the assertion races the update. Poll the actual field that's expected to change instead (see `waitUntil { ... }`).
- `IconStorageTest` also opts into `@GraphicsMode(GraphicsMode.Mode.NATIVE)` (like the screenshot tests below) even though it captures no screenshot — plain/legacy Robolectric fakes `BitmapFactory` decode results, so asserting real downsampled pixel dimensions after `IconStorage.saveImage()` needs the same real-Skia-backed decode path. `AddStationViewModelTest`'s save-triggers-file-cleanup test follows `DiscoverStationsViewModelTest`'s `awaitTrue` real-time-poll pattern (see below) since `StreamValidator.isReachable()` is on the save path; it polls the actual filesystem state (`!oldIconFile.exists()`), not `!uiState.value.isSaving`, since `isSaving` starts `false` and would satisfy an `awaitTrue` before the save coroutine ever runs.
- Screenshot tests (e.g. `StationItemScreenshotTest`) use Roborazzi over Robolectric Native Graphics (`@GraphicsMode(GraphicsMode.Mode.NATIVE)`, still `@RunWith(RobolectricTestRunner::class)` like every other test here) to catch pixel-level regressions a plain unit test can't (a shadow clipped into a rectangle, text invisible against its background) — capture with the plain `com.github.takahirom.roborazzi.captureRoboImage { content }` lambda from `roborazzi-compose`, not `createComposeRule()`/`createAndroidComposeRule<ComponentActivity>()`: this project's `AndroidManifest.xml` has no `ComponentActivity` launcher entry (and merging one in via a `src/test/AndroidManifest.xml` override still isn't enough — Robolectric's shadow `PackageManager` refused to resolve it, see git history for the abandoned attempt), whereas the lambda form registers Roborazzi's own `RoborazziActivity` into the shadow `PackageManager` at runtime (`registerRoborazziActivityToRobolectricIfNeeded()`) and needs no manifest entry or Activity-hosting rule at all. Reference images are committed to `app/src/test/screenshots/` (`roborazzi { outputDir.set(file("src/test/screenshots")) }` in `app/build.gradle` — the Roborazzi default of `build/outputs/roborazzi` would be invisible to git and to CI, since `build/` is gitignored). `./gradlew recordRoborazziDebug` re-records the baseline after an intentional visual change (review the diff before committing); `./gradlew verifyRoborazziDebug` (wired into CI's `test` job, after `./gradlew test`) fails the build on any pixel diff; plain `./gradlew test` runs the same tests but doesn't fail on a mismatch, so screenshot regressions only get caught by the dedicated verify step.
