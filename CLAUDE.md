# CLAUDE.md

Android app (Kotlin, Jetpack Compose) for playing internet radio streams (HTTP/HTTPS, incl. HLS `.m3u8`) via ExoPlayer/Media3, with local station storage in Room and background playback.

## Project layout

- Single module: `app/`. Dependency versions live in `gradle/libs.versions.toml` (Gradle auto-wires the `libs` accessor from this conventional path — no extra `settings.gradle` config needed); reference as `libs.foo.bar` / `alias(libs.plugins.foo)`, don't hardcode a `group:artifact:version` string in `build.gradle`/`app/build.gradle` again. Compose artifacts without a `version.ref` in the catalog (`androidx-compose-ui`, `-material3`, etc.) are intentionally unversioned there — they're constrained by the `androidx.compose:compose-bom` platform instead, so bumping Compose is a one-line `composeBom` version change, not touching each artifact.
- `app/src/main/java/com/urlradiodroid/`: `data/` (Room + repository), `ui/` (Compose screens + `ui/components`, `ui/playback`, `ui/theme`), `util/`.
- Lint: ktlint (`ktlintCheck` / `ktlintFormat`); `.editorconfig` allows PascalCase-looking names on `@Composable` functions (`ktlint_function_naming_ignore_when_annotated_with = Composable`). CI: `.github/workflows/ci.yml` (test + Roborazzi screenshot verify, lint, `assembleDebug` on every push/PR), `release.yml` (tag `v*` → signed `assembleRelease`, attached to a GitHub Release).
- `gradlew`/`gradlew.bat`/`gradle-wrapper.jar` are committed — use `./gradlew` locally to match CI's Gradle version.
- `app/src/main/baselineProfiles/baseline-prof.txt` + the `androidx.profileinstaller` runtime dependency give a Baseline Profile for cold start: at install time `profileinstaller` hands ART the human-readable rules in that file so it AOT-compiles the listed classes/methods instead of interpreting/JITting them on first launch. AGP 9.x reads this from the `baselineProfiles/` source-set *directory* (any number of `.txt` files merged together), not the older single `src/main/baseline-prof.txt` path — putting it there silently produces an empty profile with no error, so if `mergeReleaseArtProfile`'s output stops containing `com/urlradiodroid` entries, check the file didn't drift back to the old path. It's hand-curated (two wildcard rules covering all of `com.urlradiodroid.**`, keeping it bounded to our own code rather than guessing exact Compose/Room/Media3 internals — AGP's wildcard expansion resolves these against the real compiled classes at build time, verify with `./gradlew :app:expandReleaseArtProfileWildcards` and grep the output for `com/urlradiodroid`) rather than generated by a `androidx.benchmark.macro.junit4.BaselineProfileRule` — that needs a separate instrumented (`androidTest`) benchmark module running on a real device/emulator, which this project deliberately doesn't have (see Testing below). Re-curate by hand if `com.urlradiodroid`'s package structure changes shape significantly; there's no automated regeneration step.
- Release builds (`app/build.gradle` `buildTypes.release`) have `minifyEnabled true` + `shrinkResources true` (R8, relying on consumer proguard rules bundled in the Room/Media3/Compose AARs — `app/proguard-rules.pro` is still the stock empty template, nothing project-specific has been needed so far) and are signed via `signingConfigs.release`, which reads `RELEASE_KEYSTORE_PATH`/`RELEASE_KEYSTORE_PASSWORD`/`RELEASE_KEY_ALIAS`/`RELEASE_KEY_PASSWORD` from the environment — if `RELEASE_KEYSTORE_PATH` isn't set (e.g. a local `assembleRelease`), the signingConfig is skipped entirely and Gradle produces an unsigned `app-release-unsigned.apk` rather than falling back to debug-signing. `release.yml` decodes the keystore from the `RELEASE_KEYSTORE_BASE64` GitHub Secret to a workspace-local file (deleted with `if: always()` after the build) and supplies the other three as env vars from their own secrets — the actual `keystore/release.keystore` is gitignored and never committed; losing it means future releases can't be signed with the same key, so it (and its password) must be kept in a password manager, not just on the machine that generated it.

## Architecture conventions

- `data/RadioStationRepository` is the only class that talks to `RadioStationDao`/`AppDatabase` directly — `ViewModel`s take a `RadioStationRepository`, never an `AppDatabase`/`Context`. Created via `RadioStationRepository.create(context)`.
- Two `ViewModel` styles are in use, pick based on complexity: `MainViewModel` exposes plain `StateFlow`s and fire-and-forget `fun`s (simple CRUD/search state, no validation) plus, where a screen needs one-shot feedback, a `Channel`-backed `events: Flow<MainScreenEvent>` alongside them rather than switching the whole ViewModel to a `UiState` data class. `AddStationViewModel` uses the fuller `UiState` data class + `Channel`-backed `events: Flow<AddStationEvent>` (`SaveSucceeded`/`SaveFailed`) for one-shot UI effects (validation errors, toasts, navigating back on success) — follow whichever shape matches the screen's complexity: a bare event channel bolted onto plain `StateFlow`s for simple CRUD screens, the full `UiState`-plus-events shape for screens with form validation.
- `AddStationUiState` carries `isFavorite` even though the add/edit form has no favorite toggle of its own — `AddStationViewModel.save()` rebuilds a whole `RadioStation` from the form fields and `repository.updateStation()` is a full-row `@Update`, so any field missing from that rebuilt object gets silently reset to its data-class default. This was a real (fixed) bug: before `isFavorite` was added to `AddStationUiState`/loaded in `init`/passed through in `save()`, editing a favorited station silently un-favorited it. `genre` follows the same "load in init, thread through save()" shape, but it *is* directly editable via `onGenreChange`. The general lesson — any new `RadioStation` field must be added to `AddStationUiState` and both `save()` branches (new + edit), not just the entity — is the same trap a future field addition here would hit again.
- Deleting a station is immediate but undoable, not gated behind a confirmation dialog: `MainViewModel.deleteStation(station)` deletes from the DB right away, reloads the list, then emits `MainScreenEvent.StationDeleted(station)`; `MainScreen` collects `viewModel.events` and shows a `Snackbar` ("Station deleted" / "Undo") via `SnackbarHostState` — if the user taps Undo before the snackbar times out, `MainViewModel.undoDelete(station)` re-inserts the same `RadioStation` object (including its original `id`), which SQLite accepts as an explicit rowid for an `autoGenerate` primary key as long as nothing reused it since, so the restored station reappears in its original position (`getAllStations()` orders by `id ASC` within the same-favorite group) rather than jumping to the end of the list.
- `RadioPlaybackService` (a `MediaLibraryService`, see Android Auto below) exposes a `playbackSnapshot: StateFlow<PlaybackSnapshot>` (isPlaying, isBuffering, currentMediaId, hasTimeshift, isAtLive, trackTitle, sleepTimerEndAtMs) as the single source of truth for reactive UI state — screens collect this instead of polling the service. It delegates to two extracted collaborators: `playback/TimeshiftController` (buffer file lifecycle + seek math for the rewind/live feature) and `playback/PlaybackStateStore` (SharedPreferences-backed; remembers the last-started station so `onStartCommand` can resume playback if the system kills and restarts the service with a null intent — this is what makes background playback survive process death, not just app backgrounding).
- Any `Activity` that binds `RadioPlaybackService` and passes the instance into a `@Composable` (`MainActivity`, `PlaybackActivity`) must hold it as `mutableStateOf<RadioPlaybackService?>`, not a plain `var`. `ServiceConnection.onServiceConnected` fires asynchronously after the first composition; a plain field mutation there is invisible to Compose, so the composable (and anything derived from it via `playbackSnapshot`, e.g. `isPlaying`/track title/sleep timer) stays stuck at its initial `null` forever.
- Reconnect-on-network-loss uses capped exponential backoff (`RadioPlaybackService.retryDelayMs`: 2s/4s/8s/16s, capped at 30s, `MAX_RETRY_COUNT = 5`) rather than retrying forever — after the limit it surfaces "Connection failed" and stops instead of hammering the stream.
- Station icons (`RadioStation.customIcon`, doc comment: "Emoji string or image file path") default to auto-generated from the name/URL via `util/EmojiGenerator` (keyword + hash fallback) — except for Discover-added stations with a directory favicon, backfilled asynchronously via `RadioBrowserApi.downloadFavicon()`/`IconStorage.saveImageBytes()`, see Station discovery below — but can be overridden via the picker dialog (`ui/components/StationIconPickerDialog.kt`, opened by tapping the circular preview at the top of `AddStationScreen`) — "Choose Emoji" picks from `EmojiGenerator.pickerEmojis`, "Upload Image" launches the system photo picker (`ActivityResultContracts.PickVisualMedia`, no runtime permission needed) and `util/IconStorage.saveImage()` downscales the picked image to at most 512px and copies it as a JPEG into `filesDir/station_icons/`, "Remove Icon" clears it back to the auto-generated fallback. A leading `/` is what `IconStorage.isImagePath()` uses to tell a stored file path apart from an emoji string (no emoji starts with that character), and every render site (`StationItem`, `AddStationScreen`'s preview — via the shared `rememberStationIconBitmap()` composable in `ui/components/StationIcon.kt`) falls back to the emoji path if the file is missing/corrupt. **`NowPlayingBottomBar`'s two mini-player card variants deliberately don't show it** — both used to render the same `rememberStationIconBitmap()`/emoji-fallback icon in a 56dp circle at the start of the row, but it was dropped so the station name (already fighting for room against the play/pause, rewind, and live-indicator buttons at that card's fixed 80dp height) gets that width back instead; the full-screen `PlaybackScreen` still shows the emoji (not `customIcon`, per the "not yet wired up" note above) at a larger size where the tradeoff doesn't apply. `AddStationViewModel` only deletes the *replaced* icon file, and only once `save()` actually commits (compares the freshly-loaded `originalCustomIcon` against the session's final value) — it deliberately does not clean up on cancel/back or on intermediate re-picks before saving, so an abandoned edit can leak a small orphaned JPEG; accepted as a minor, bounded cost rather than adding cleanup-on-cancel plumbing. **Not yet wired up:** `PlaybackScreen` (full-screen player) and `RadioPlaybackService`'s notification large-icon still only show the auto-generated emoji — neither has ever taken `customIcon` into account (this predates the picker), and threading a custom image through the notification/media-session pipeline (`EXTRA_STATION_ICON` on every intent that starts/resumes playback, `PlaybackStateStore` persistence, decoding a bitmap in `getCurrentLargeIcon`) is a larger, separate change.
- `ui/theme/Dimens.kt` holds the design tokens: `Spacing` (4/8/16/24dp scale — only use where a padding/gap value already matches one of these exactly, don't force-fit one-off values like a 20dp chip padding into the scale) and `AppShapes` (a Material3 `Shapes` wired into `MaterialTheme(shapes = AppShapes)` in `Theme.kt`: `extraSmall`=8dp for icon-sized controls, `medium`=16dp for buttons/fields/chips, `large`=24dp for primary cards, `extraLarge`=40dp for the pill-shaped mini-player cards). Reference them as `MaterialTheme.shapes.large` / `Spacing.md` instead of new `RoundedCornerShape(Ndp)`/raw `.dp` literals so radii and rhythm stay consistent across screens.
- `URLRadioDroidTheme` is always-dark by design (no system-theme following, no light `ColorScheme`, no `dynamicColor` param) — `card_surface`/`text_primary`/etc. in `Color.kt` are hardcoded dark values that assume this, so a default Material component left without explicit `colors` (a `DropdownMenu`, a `TextField`) would go invisible-on-light if the theme ever did follow the system. Don't reintroduce `isSystemInDarkTheme()` or a light `ColorScheme`; if a screen needs to look different, do it with an explicit param, not by resurrecting system-theme following.
- When a `Modifier` chain combines a rounded `shape` with `.clickable()`/ripple (or is passed as a `Card`'s `modifier`), the shape-clip must come before the click handler in the chain — otherwise the ripple/press-state paints the full rectangular bounds and bleeds past the rounded corners. Prefer `Card(onClick = ..., border = BorderStroke(...))` (shape-aware internally) over chaining `Modifier.border()`/`.clickable()` onto a `Card`'s incoming `modifier`; for plain `Box`/`Row` click targets, order it `.clip(shape).clickable(...)`.
- `util/AppShortcuts.kt` publishes up to 2 dynamic App Shortcuts (long-press the launcher icon): the last-played station (from `playback/PlaybackStateStore`) and the top favorite (`stations.firstOrNull { it.isFavorite }`, same ordering as the DB query), deduped to one if they're the same station. `AppShortcuts.refresh(context, stations)` is called from a `LaunchedEffect(allStations)` in `MainScreen` — it re-fires on every `loadStations()` (including plain `onResume()`), which is also the only point the last-played shortcut picks up a station started elsewhere in the same session; there's no dedicated observer on `PlaybackStateStore`, so a shortcut can lag until the user next opens `MainActivity`, same "good enough" freshness bar as this codebase's other minor accepted gaps (e.g. the icon-picker's orphaned-file cleanup above). Both shortcuts always show the auto-generated emoji, never `RadioStation.customIcon` — the last-played entry only has a name/URL from `PlaybackStateStore`, not a full `RadioStation`, so there's nothing to read a custom icon off of; matches the existing scope boundary where the notification/`PlaybackScreen` also don't show `customIcon`. Each shortcut's intent targets `PlaybackActivity` with `EXTRA_AUTO_PLAY = true`, which makes `handleIntent()` call `togglePlayback()` immediately instead of just opening the screen and waiting for a tap — **this requires `PlaybackActivity` to be `android:exported="true"`** (a hard OS requirement: `ShortcutManagerCompat.setDynamicShortcuts` throws if a shortcut's target activity isn't exported), so any other installed app can now also launch it with arbitrary `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` extras and make this app start playing an arbitrary stream URL — accepted as low-severity (no data access, no permission escalation, same trade-off most media-player "quick play" shortcuts/widgets make) rather than adding a signature-permission gate.
- **Wake-up alarm** (`ui/AlarmScreen.kt`'s `AlarmActivity`, opened from MainScreen's overflow menu) is a single daily alarm, not a list of alarms — a deliberately smaller scope than a full clock app, matching "wake up to radio" rather than a general alarm clock. It's a fundamentally different mechanism from the sleep timer (`RadioPlaybackService.setSleepTimer`, a `Handler.postDelayed` that only exists while the service process is alive): the alarm must fire even if the app/service isn't running at all, so it's built on `AlarmManager.setAlarmClock` (`ui/AlarmScheduler.kt`) rather than reusing any sleep-timer plumbing. `setAlarmClock` — not `setExactAndAllowWhileIdle` — is the deliberate choice because apps that show a user-visible alarm (this one puts the status-bar alarm-clock icon up, via `AlarmClockInfo`) are *documented* as exempt from the `SCHEDULE_EXACT_ALARM`/`USE_EXACT_ALARM` permission dance. **That exemption isn't guaranteed in practice**, though — `setAlarmClock()` has been observed throwing `SecurityException` on some devices/OS builds regardless, which crashed `BootReceiver` outright (a manifest-registered receiver has no UI to recover from an uncaught exception). `AlarmScheduler.schedule()` now returns `Boolean` rather than being `Unit`: it checks `AlarmManager.canScheduleExactAlarms()` first (API 31+ only; always true below that) and also wraps the actual `setAlarmClock()` call in `try/catch(SecurityException)` as a last-resort safety net, so it can never throw. `AndroidManifest.xml` declares `SCHEDULE_EXACT_ALARM` (needed for `canScheduleExactAlarms()`/the grant flow to have any effect at all) — this is a special, install-time-declared permission the user grants via Settings, not a runtime `ActivityCompat.requestPermissions` dialog. `AlarmScreen`'s save button is the only call site with UI to react to a `false` result: it shows `alarm_permission_needed` and launches `Settings.ACTION_REQUEST_SCHEDULE_EXACT_ALARM` (API 31+) so the user can grant "Alarms & reminders" access; `BootReceiver`/`AlarmReceiver` just discard the `Boolean` (no UI to react from — silently not rescheduling is the graceful-degradation outcome instead of a crash). State (`enabled`, `hour`, `minute`, `stationName`, `streamUrl`) lives in `ui/playback/AlarmStateStore.kt` (SharedPreferences, same shape as `PlaybackStateStore`) rather than Room, since it's a single row read by two `BroadcastReceiver`s that shouldn't need a DB open. `AlarmReceiver` fires at the scheduled time (`startForegroundService` with the same `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` extras `RadioPlaybackService` already accepts elsewhere), then immediately reschedules itself for the same time tomorrow — this is what makes it a *daily* alarm without a day-of-week picker UI. `BootReceiver` (`RECEIVE_BOOT_COMPLETED`) exists because `AlarmManager` alarms are cleared on reboot; it just re-schedules from the saved state if `enabled`. `AlarmScheduler.nextTriggerMillis(hour, minute, nowMillis)` is `internal` and covered directly by `AlarmSchedulerTest`'s pure-function tests (today-vs-tomorrow decision, including the exactly-now edge case) — that file also now needs `@RunWith(RobolectricTestRunner::class)`/`@Config(sdk = [29])` at the class level (with per-method `@Config(sdk = [31])` overrides for the permission-gate tests, using `ShadowAlarmManager.setCanScheduleExactAlarms()`) purely for the `schedule()` coverage added alongside it, even though `nextTriggerMillis` itself is still plain `java.util.Calendar` math with no Android framework dependency.
- Both `Switch`es in the app (`SettingsScreen`'s metered-warning toggle, `AlarmScreen`'s enable toggle) share one explicit `SwitchDefaults.colors(...)` (checked: `text_primary` thumb on a `glass_accent` track; unchecked: `text_hint` thumb on a `card_surface` track with a `card_border` outline) rather than leaving most of it at the Material3 default. Leaving only `checkedThumbColor` overridden (the original code) left the checked *track* at the theme's default `colorScheme.primary` — `md_theme_dark_primary` (`0xFFFFB870`), a warm amber almost identical to `glass_accent` (`0xFFFFAF5F`) — so a checked switch's thumb was nearly invisible against its own track, and both nearly blended into this app's warm-amber background gradient. If a 3rd `Switch` is ever added, reuse this same explicit color set rather than the bare default.
- **Settings screen** (`ui/SettingsScreen.kt`'s `SettingsActivity`, opened from MainScreen's overflow menu, same "standalone `ComponentActivity` + its own `ViewModel`" shape as `AlarmActivity`/`AlarmViewModel`) is where "Export stations"/"Import stations" live now — they used to be overflow-menu items directly on `MainScreen` (the export/import JSON logic itself, `StationShare.share()`/`RadioStationRepository.importStationsFromJson()`, is unchanged, just relocated into `SettingsScreen`'s composable and `SettingsViewModel`, which now holds the two thin `exportStationsJson()`/`importStationsJson()` delegates `MainViewModel` used to). The one setting so far is a metered-connection playback warning: `ui/playback/SettingsStore.kt` (SharedPreferences, same shape as `PlaybackStateStore`/`AlarmStateStore`) persists a single `warnOnMeteredConnection: Boolean` (default `true`). The check itself lives in `MainScreen` — a private `isMeteredConnection(context)` (`ConnectivityManager.isActiveNetworkMetered`) gates only the direct play-button tap on a `StationItem` in the station list (`onStationItemPlayClick`, wired through a `pendingMeteredStation` state + `AlertDialog` with "Play anyway"/"Cancel") — deliberately **not** the mini player/full-screen `NowPlayingBottomBar` (`onPlayStationWithState` is still called directly there, ungated), the home screen widget, Android Auto's browse tree, or the wake-up alarm, since those are either already-in-progress playback switches or automatic/background starts with no good place to show a blocking confirmation dialog.

## Localization

- `res/values/strings.xml` is the source of truth (English); translations live in `res/values-<lang>/strings.xml` with the exact same set of `name` keys — no key may be added/removed/renamed in one without mirroring the change in the other, since Android silently falls back to the default `values/` string for any key missing in a locale rather than erroring. Translated locales: `values-ru` (Russian), `values-es` (Spanish), `values-zh-rCN` (Simplified Chinese). Format-argument placeholders (`%1$s`, `%1$d:%2$02d`, etc.) must be preserved verbatim — same positions, same conversion types — since callers pass args positionally via `stringResource(R.string.x, arg1, arg2)`. `app_name` is a brand name and stays untranslated in every locale.
- **Any feature that adds/renames/removes a string in `res/values/strings.xml` must make the matching edit in all three `values-<lang>/strings.xml` files in the same change** — don't leave a feature English-only pending a later translation pass. Verify key parity before considering the change done: `diff <(grep -o 'name="[^"]*"' app/src/main/res/values/strings.xml | sort) <(grep -o 'name="[^"]*"' app/src/main/res/values-<lang>/strings.xml | sort)` should be empty for each locale.

## Database & migrations

- `RadioStation` has unique indices on `name` and `streamUrl` (DB-level, added in schema v3) on top of the pre-existing app-level `isNameTaken`/`isUrlTaken` checks — defense in depth against any insert path that bypasses the UI checks. `AppDatabase` is now on `exportSchema = true` (schema JSONs checked into `app/schemas/`, `room.schemaLocation` set via the `ksp {}` block in `app/build.gradle`) — **schema changes must ship a real `Migration` and a matching schema snapshot, not rely on `fallbackToDestructiveMigrationFrom`** (only allowed for pre-2.0 installs jumping from version 1, since there's no v1 schema snapshot to migrate from safely). `AppDatabase.MIGRATION_2_3` is the template to follow: dedupes any pre-existing `name`/`streamUrl` collisions (`DELETE ... WHERE id NOT IN (SELECT MIN(id) ... GROUP BY ...)`, keeping the oldest row) before creating the unique indices, so upgrading never crashes or silently drops the wrong rows even if duplicates somehow predate the constraint.
- `AppDatabaseMigrationTest` verifies migrations against a hand-built copy of the pre-migration schema (kept in sync with `app/schemas/.../2.json`, `.../3.json`) rather than `MigrationTestHelper`, since that needs an `androidTest` source set with schema assets + instrumentation, which this project deliberately doesn't have (unit-tests-only, see Testing below) — needs `kspTest 'androidx.room:room-compiler:2.8.4'` so KSP generates Room impl classes for the test-only entities/DB it defines. Each migration gets its own `LegacyVN...` entity/DAO/DB snapshot in that file (not the real `RadioStation`/`AppDatabase`, which already carry later columns/indices and would make the "legacy" database already match the target schema, defeating the point of the test) — `AppDatabase.MIGRATION_3_4` (adds `isFavorite`) is the template for a plain `ALTER TABLE ... ADD COLUMN` migration, versus `MIGRATION_2_3`'s dedupe-then-constrain pattern.
- `RadioStation.isFavorite` (schema v4) drives the favorites/pinning feature: `RadioStationDao.getAllStations()` orders `isFavorite DESC, id ASC` so pinned stations float to the top while preserving insertion order within each group — this is the single source of truth for list ordering, not a client-side sort, so it applies everywhere `getAllStations()` is used (list, export). Toggled via `MainViewModel.toggleFavorite()` → `RadioStationDao.setFavorite()`, surfaced as a star icon on `StationItem` (next to the play button, not behind the edit/share/delete swipe reveal, since favorite status needs to be visible at a glance and togglable in one tap). Included in the backup JSON shape (`StationBackupJson`) and read back via `optBoolean("isFavorite", false)` so older backups without the field still import cleanly.
- `RadioStation.genre` (schema v5, nullable `TEXT`, plain `ALTER TABLE ... ADD COLUMN` migration — `MIGRATION_4_5`, same template as `MIGRATION_3_4`) is freeform text, not a fixed enum/tag set: Discover-added stations get it prefilled verbatim from the Radio Browser directory's own comma-separated `tags` field (`DiscoverStationsViewModel.addStation`, see Station discovery below), manually-added stations get an optional free-text field in `AddStationScreen`/`AddStationViewModel`, and `MainViewModel.filteredStations`'s search matches it alongside name/URL. Shown in `StationItem`'s subtitle in place of the raw stream URL when present (the URL isn't very useful to glance at; matches the same "show something more meaningful than the URL" instinct already behind Discover's result subtitle). Included in the backup JSON shape the same way `isFavorite` is (`optString("genre")`/JSON-null-safe, older backups without the field import with `genre = null`).
- `RadioStation.isHls` (schema v6, `Boolean` defaulting to `false`, plain `ALTER TABLE ... ADD COLUMN` migration — `MIGRATION_5_6`, same template as `MIGRATION_3_4`/`MIGRATION_4_5`) is a known-HLS hint sourced from the Radio Browser directory's own `hls` flag (`DiscoverStationsViewModel.addStation` sets it from `RadioBrowserStation.hls`; manually-added stations always get `false`, since `AddStationScreen` has no toggle for it — same "carried through unedited, no UI" pattern as `isFavorite` in `AddStationUiState`). It exists to make `RadioPlaybackService.isHlsUrl()` more reliable than its plain `"m3u8" in url` heuristic for Discover-added streams whose URL doesn't happen to contain that substring; see Station discovery below for how it flows into actual playback.
- `RadioStation.radioBrowserUuid` (schema v7, nullable `TEXT`, plain `ALTER TABLE ... ADD COLUMN` migration — `MIGRATION_6_7`, same template as the columns above) is the Radio Browser directory's `stationuuid`, filled in the same "Discover sets it, manual adds get `null`, carried through unedited in `AddStationUiState`" way as `isHls`. It exists purely so `RadioPlaybackService` can register a play as a "click" with the directory on actual playback — see Station discovery below.

## Backup: export / share / import

- Backup JSON shape (`{name, streamUrl, customIcon, isFavorite, genre, isHls, radioBrowserUuid}[]`) is centralized in `data/StationBackupJson.kt`, used both by `RadioStationRepository.exportStationsToJson()` (all stations, from DB) and by per-station sharing (serializes a single in-memory `RadioStation`, no DB round-trip). `RadioStationRepository.importStationsFromJson()` reads the same shape (JSON via `org.json`) and is wired to `SettingsScreen`'s "Import stations" (SAF `OpenDocument`) — see Settings in Architecture conventions above.
- "Export stations" (`SettingsScreen`, all stations) and per-station "Share" (swipe action on `StationItem`, one station) both go through `util/StationShare.kt`, which sends the JSON as a file attachment via `Intent.ACTION_SEND` (so it can go straight to Telegram/WhatsApp/email/etc., not just to a chosen SAF location) — the file name is the station name for per-station shares. Needs a `FileProvider` (declared in `AndroidManifest.xml`, paths in `res/xml/file_paths.xml`, authority hardcoded as `com.urlradiodroid.fileprovider` in `StationShare` — must stay in sync with `applicationId` since there's only one build variant). `testOptions.unitTests.includeAndroidResources = true` in `app/build.gradle` is required for Robolectric to see manifest-declared components (providers, etc.) in tests — without it `FileProvider` can't find its meta-data at all. Note: `FileProvider` statically caches its resolved roots per authority for the process lifetime, so `StationShareTest` deliberately uses a single test method rather than one Activity/cacheDir per `@Test`.
- Currently playing track title (shown in `PlaybackScreen`, sourced from ICY in-band metadata parsed by `StreamRecorder`) has a copy-to-clipboard button next to it, using Compose's `LocalClipboardManager` — no `ClipboardManager` system service needed. Both `PlaybackScreen`'s track-title `Text` and `NowPlayingBottomBar`'s mini-player one (`MiniPlayerCardFull` — the "full" active-card variant; the sibling `MiniPlayerCardPreview` never shows a track title at all) use `Modifier.basicMarquee()` with `maxLines = 1`, since track titles routinely overflow the available width and previously just got silently clipped/ellipsized; `PlaybackScreen`'s also gained an explicit `maxLines = 1` it was missing entirely (it could wrap to multiple lines before).

## Station discovery & stream validation

- `data/RadioBrowserApi.kt` is a minimal client for the [Radio Browser API](https://api.radio-browser.info/) directory (constructs its own `OkHttpClient`, parses with `org.json`, not injected as a `Retrofit`/DI dependency — same low-ceremony style as `StationBackupJson`). It hits `https://all.api.radio-browser.info/` by default, which round-robins across every community mirror, so no manual DNS-SRV server discovery is needed. `baseUrl`/`client` are constructor params purely so tests can point it at a `MockWebServer`.
- `ui/DiscoverStationsViewModel` + `ui/DiscoverStationsScreen.kt` (`DiscoverStationsActivity`, launched from a standalone `AssistChip` at the top of MainScreen's list, not the overflow (⋮) menu — it's a headline feature and that row otherwise had dead space above the search field) let the user search the directory by name/genre/country/language and add results straight into the library. `DiscoverSearchMode`'s `LANGUAGE` mode (→ `RadioBrowserApi.SearchBy.LANGUAGE`, param `language`) follows the exact same shape as `NAME`/`GENRE`/`COUNTRY` — a `SearchModeChip` + a hint string, both listed in a `Row` wrapped in `Modifier.horizontalScroll(rememberScrollState())` since 5 chips no longer reliably fit `fillMaxWidth()` on narrow screens. Chip order is deliberately `NAME, NEARBY, GENRE, COUNTRY, LANGUAGE` — `NEARBY` sits 2nd (right after the default mode), not last, since it's a headline mode in its own right, not an afterthought. Typing is debounced 400ms before calling the API (`SEARCH_DEBOUNCE_MS`). Unlike `AddStationActivity`, this screen doesn't `finish()`/return a result after each add — the user browses and adds multiple stations in one visit, so `MainActivity`'s `discoverStationsLauncher` reloads the station list unconditionally on return (no `RESULT_OK` gate) rather than following `addStationLauncher`'s pattern. Name collisions on add are auto-disambiguated (`"Station (2)"`, `"Station (3)"`, ...) rather than blocking the add or silently overwriting. `DiscoverResultCard`'s station-name `Text` uses `Modifier.basicMarquee()` (`maxLines = 1`, no explicit `TextOverflow` — marquee supersedes ellipsis truncation) since result names routinely don't fit the available width once the vote/SSL-warning/homepage/add-button icons are accounted for; the card's inter-element `Arrangement.spacedBy` and horizontal padding were also tightened from `Spacing.md` to `Spacing.sm` to free up more room for the name.
- `util/StreamValidator.kt` probes a URL before `AddStationViewModel.save()` commits it (HEAD first, falling back to GET for Icecast/Shoutcast mounts that reject HEAD with 405) — it closes the response as soon as headers arrive without reading the effectively-infinite stream body. This runs after the existing name/URL-taken checks (cheap DB lookups first, network probe only if those pass) and surfaces `error_stream_unreachable` on failure instead of letting a bad URL surface as "Connection failed" the first time the user hits play.
- `RadioBrowserStation.hls` (parsed from the directory's `hls` 0/1 field, defaulting to `false` when absent) is threaded through to `RadioStation.isHls` on insert (`DiscoverStationsViewModel.addStation`) and from there into actual playback: `RadioPlaybackService.isHlsUrl(url, knownHls)` prefers `knownHls` over its `"m3u8" in url` heuristic when non-null. Since `onStartCommand`/`startPlayback`/`applyPlayback` only ever receive a bare stream URL (every caller — `MainActivity`, `PlaybackActivity`, the widget, the alarm, App Shortcuts — passes `EXTRA_STATION_NAME`/`EXTRA_STREAM_URL` strings, never a full `RadioStation`), `onStartCommand` resolves the hint itself via `repository.getStationByUrl(streamUrl)` in a `serviceScope.launch` before calling `startPlayback` — a small extra Room query on every playback start rather than threading a new intent extra through every one of those call sites. Android Auto's `playFromBrowseTree`/`onAddMediaItems` skip that DB round-trip entirely and pass `station.isHls` straight from the already-in-memory `cachedStations`. A network-loss retry (`tryResumePlaybackAfterNetworkRestored`/`retryPlaybackIfPending`) reuses the hint from `currentKnownHls` (cached alongside `currentStreamUrl` in `applyPlayback`) rather than re-resolving it, so a retry of the same stream can't flip-flop the HLS decision.
- `RadioBrowserApi.registerClick(uuid)` (`GET /json/url/{uuid}`) is how the directory expects a play to be reported so its own `clickcount` popularity metric accounts for this app's plays, not just the directory's own web/other clients — a separate number from `votes` (which `search()` actually sorts results by, see below). It's deliberately fire-and-forget: swallows `IOException` internally (network failure, unreachable mirror) rather than throwing, since a telemetry call must never block or fail a play. `RadioStation.radioBrowserUuid` reuses the exact same "resolve by URL in `onStartCommand`'s `serviceScope.launch`, or read straight off `cachedStations` in `playFromBrowseTree`" plumbing `isHls`/`knownHls` already established just above — the two hints (`isHls`, `radioBrowserUuid`) are resolved from the same `repository.getStationByUrl(streamUrl)` call. Only fired from the direct-play branch of `onStartCommand` (a genuine new `EXTRA_STREAM_URL` intent) and from `playFromBrowseTree` (Android Auto) — deliberately **not** from the process-restart-resume branch (`playbackStateStore.restore()`), since resuming after the system killed and restarted the service isn't a new user-initiated play and would otherwise double-count or fire on every cold start.
- `RadioBrowserStation.countryCode` (parsed from the directory's `countrycode` ISO 3166-1 alpha-2 field) feeds `util/CountryFlagEmoji.from(code)`, a pure function that converts a 2-letter code to its flag emoji via the fixed Unicode regional-indicator-symbol offset (`0x1F1E6` = `'A'`'s indicator symbol) — no network call or bundled flag asset needed, and it returns `null` for anything that isn't exactly 2 ASCII letters (blank/absent code, malformed data) so callers can fall back to omitting it. `DiscoverStationsScreen`'s private `stationSubtitle()` shows just the bare flag emoji when available (`"🇩🇪 pop,talk • 128 kbps MP3"`) — no country name text alongside it, since the flag alone reads faster and the name was redundant screen-width cost; the country part is dropped entirely (not a text fallback) when no flag can be derived. `RadioBrowserStation.country` (the display name) is still parsed and stored on the struct, just no longer rendered in the subtitle. `RadioBrowserStation.codec` (the directory's `codec` field, e.g. `MP3`/`AAC+`/`OGG`) is folded into that same subtitle's bitrate segment (`"128 kbps MP3"`) rather than a separate `•`-joined part, since the two are naturally one unit; also presentation-only, not persisted.
- `RadioBrowserStation.votes` feeds a small popularity badge (a star icon + `util/VoteCountFormatter.format(votes)`, e.g. `"1.5k"`/`"2.5M"`) shown on `DiscoverResultCard` next to the add button when `votes > 0` — `search()`'s `order=votes` sorts results by this exact same number, so the visible ranking matches what the star badge implies (previously sorted by `clickcount`, a different metric the badge doesn't show, which read as "random" order relative to the visible vote counts). `registerClick()` still contributes to the directory's separate `clickcount` metric, just no longer the one this app's own search results are ranked by.
- `RadioBrowserStation.homepage` (the station's own website, when the directory has one) surfaces as an `Icons.AutoMirrored.Filled.OpenInNew` icon button on `DiscoverResultCard` (shown only when non-blank) that opens it via a plain `Intent.ACTION_VIEW`, caught against `ActivityNotFoundException` (a malformed homepage URL with no app able to handle it is a silent no-op, not a crash) — presentation-only, not persisted, same as `codec`/flag emoji above.
- `RadioBrowserStation.favicon` (the directory's URL to the station's real logo) feeds the existing custom-icon pipeline as an auto-fill source for Discover-added stations, alongside `util/EmojiGenerator`: `RadioBrowserApi.downloadFavicon(url)` fetches the image bytes (returns `null` on any failure — bad URL, unreachable host, non-2xx — never throws, so a broken favicon can't block adding the station) and `util/IconStorage.saveImageBytes(context, bytes)` (new sibling to `saveImage(context, uri)`, sharing the same downscale-to-512px-then-JPEG `persistDownscaled()` internals — the only difference is the source: already-downloaded bytes vs. a picker `Uri`) persists it the same way a manually-picked photo would. This happens in `DiscoverStationsViewModel.downloadAndSetFavicon()`, fired **fire-and-forget** from `addStation()` (a `launch` child of the same `viewModelScope.launch` that just inserted the station) rather than blocking the "Added" checkmark on a network round-trip — the station shows the auto-generated emoji immediately and its `customIcon` gets backfilled via `repository.updateStation()` once the download finishes, or never if it fails or the ViewModel's scope is gone first (same accepted "good enough, no cleanup plumbing" tradeoff as the icon-picker's orphaned-file case above). `DiscoverStationsViewModel` now takes an `appContext: Context` constructor param (stored from `context.applicationContext` in `provideFactory`, never a raw Activity context, so it can't leak one) purely for this — the only ViewModel in the codebase that holds a `Context`, since `IconStorage` needs one and `RadioStationRepository` doesn't expose a byte-saving equivalent.
- **"Near me" search** (`DiscoverSearchMode.NEARBY`, the 2nd `SearchModeChip` with a location-pin icon — see chip order above) is a fundamentally different mode from `NAME`/`GENRE`/`COUNTRY`/`LANGUAGE`: it has no text query, so `DiscoverStationsScreen` hides the `TextField` entirely when this mode is selected and instead drives a one-shot permission-then-search flow. If the search fails because location services (GPS) are off, the app can't observe that turning on directly — the user has to leave to system Settings to enable it — so `DiscoverStationsScreen` installs a `LifecycleEventObserver` (`LocalLifecycleOwner`, wrapped in `DisposableEffect`) that retries `requestNearbySearch()` on every `ON_RESUME` while `mode == NEARBY` and the last result was specifically `discover_location_unavailable`; this is what makes coming back from Settings with GPS now on "just work" instead of requiring the user to re-tap the Nearby chip. The observer reads `uiState` through `rememberUpdatedState` (not a captured `val`) since the `DisposableEffect` itself is keyed only on `lifecycleOwner` and so isn't re-created on every state change — without that, the resume callback would see a stale, possibly-permanently-`false` snapshot of the retry condition. `RadioBrowserApi.searchNearby(latitude, longitude, radiusMeters, limit)` hits the same `json/stations/search` endpoint as `search()` but with `geo_lat`/`geo_long`/`geo_distance` and `order=distance` instead of a text param — `RadioBrowserStation.distanceKm` is populated only for this call (parsed from the response's `distance` meters field when present, `null` otherwise) and shown as a `"📍 2.3 km"` prefix in `stationSubtitle()`. The 50km search radius (`NEARBY_RADIUS_METERS`) is hardcoded, not user-configurable.
- `util/LocationProvider.kt` wraps the plain `android.location.LocationManager` (not play-services-location's `FusedLocationProviderClient` — deliberately avoids a new Gradle/Google-Play-Services dependency, matching `RadioBrowserApi`/`StreamValidator`'s own "build our own client" style) to resolve a single, one-shot fix for `getCurrentLocation()`: it prefers a cached `getLastKnownLocation()` over waiting for a fresh `requestSingleUpdate()`, and never requests continuous updates or persists a coordinate anywhere. `hasPermission()`/`getCurrentLocation()` both gate on `ACCESS_COARSE_LOCATION` only (not `ACCESS_FINE_LOCATION`) — city/neighborhood-level precision is enough for a station-radius search, and it's a real, deliberate part of the privacy tradeoff ROADMAP.md flagged before this was built.
- The permission flow itself lives entirely in `DiscoverStationsScreen` (not the ViewModel, which has no Android permission APIs to call): tapping the "Near me" chip checks `ContextCompat.checkSelfPermission` directly; if ungranted, an `AlertDialog` rationale (explains the location is sent only for this one search and never stored) gates a `rememberLauncherForActivityResult(ActivityResultContracts.RequestPermission())` launch, same pattern `AddStationScreen`'s photo picker already established for a different permission-adjacent system picker. On grant, the screen calls `viewModel.searchNearby()` directly; on denial (either from the system dialog or the rationale's Cancel button), `viewModel.onLocationPermissionDenied()` sets `DiscoverStationsUiState.locationPermissionDenied`, shown as an inline hint under the chip row rather than blocking the rest of Discover.
- Testing `LocationProvider` needs two non-obvious Robolectric fixes documented in `LocationProviderTest`'s own class doc: (1) permission grant/deny must go through `Shadows.shadowOf(application).grantPermissions()/denyPermissions()` (`ShadowContextWrapper`, inherited by `ShadowApplication`) — the seemingly-obvious `ShadowApplicationPackageManager.grantRuntimePermission()` is both `protected` (needs `ReflectionHelpers` to even call) *and*, empirically, doesn't affect `ContextCompat.checkSelfPermission()`'s result under this Robolectric version anyway; (2) `ShadowLocationManager.simulateLocation()` delivers to a `requestSingleUpdate()` listener by posting to the `Looper` it was registered against (here `Looper.getMainLooper()`), and Robolectric's main looper is paused by default, so the callback never fires without an explicit `Shadows.shadowOf(Looper.getMainLooper()).idle()` after `simulateLocation()` — a real 5-second internal timeout otherwise silently eats the failure, since the test just sees `null` rather than an exception.
- `RadioBrowserStation.sslError` (parsed from the directory's `ssl_error` 0/1 field, defaulting to `false` when absent) surfaces as a small `Icons.Default.Warning` icon (tinted `MaterialTheme.colorScheme.error`, content description `discover_ssl_warning`) on `DiscoverResultCard` next to the homepage link — a non-blocking warning rather than hiding the result outright, since the directory's own HTTPS check failing doesn't necessarily mean every client (including this app's ExoPlayer-based playback) will also fail on it. Presentation-only, not persisted on `RadioStation`, same as `codec`/`countryCode`/flag emoji above.

## Android Auto

- `RadioPlaybackService extends MediaLibraryService` (not the plain `MediaSessionService` this used to be) so the station list can be browsed from a car head unit. The manifest intent-filter action changed from `androidx.media3.session.MediaSessionService` to `androidx.media3.session.MediaLibraryService` (plus `android.media.browse.MediaBrowserService` for legacy `MediaBrowserServiceCompat` clients) — `MediaLibraryService.SERVICE_INTERFACE` really is a different string from `MediaSessionService.SERVICE_INTERFACE`, not a supertype/subtype the old filter would still match, so keeping the old action would leave Auto unable to find the service at all. The service is also now `android:exported="true"` (previously `false`) since Auto binds to it from another process, and `res/xml/automotive_app_desc.xml` (`<automotiveApp><uses name="media"/></automotiveApp>`, referenced via a `com.google.android.gms.car.application` `<meta-data>` on `<application>`) is the standard descriptor that makes a media app eligible to show up in Auto at all.
- **`onBind()` bug fixed as part of this**: it used to unconditionally `super.onBind(intent)` and then discard the result, always returning the app's own `LocalBinder`. `MediaSessionService`/`MediaLibraryService.onBind()` actually returns the *real* session binder when the incoming intent's action matches the media-session interface (external clients: Auto, Assistant, legacy browsers) and `null` for anything else (including our own Activities' plain explicit-component bind, which has no action set) — so the old code was silently handing Auto/Assistant an unrelated `Binder` instead of the session binder, which would have broken their connection outright the moment anything external tried to bind. Now: `return super.onBind(intent) ?: binder`.
- A `MediaLibrarySession` now has to exist immediately in `onCreate()` (`buildMediaSession(streamUrl = null)`, called right after `initializePlayer()`) rather than being built lazily on the first `startPlayback()` call like before — `onGetSession()` must return non-null for Auto to browse the library even before anything has ever played in this process. `buildMediaSession(streamUrl: String?)` still releases and rebuilds the session on every real `startPlayback()` (unchanged behavior, needed so the session's `sessionActivity` PendingIntent carries fresh station extras), it just also accepts `null` for that one initial no-station build, in which case the open-app intent carries no extras and relies on `PlaybackActivity` reading the live station from the bound service once it connects.
- The browse tree is flat (root → all stations, `isFavorite DESC, id ASC`, same ordering as everywhere else `RadioStationDao.getAllStations()` is used) via `MediaLibrarySessionCallback.onGetLibraryRoot`/`onGetChildren`/`onGetItem`. Station media items use `mediaId = streamUrl` (matching the scheme already used for `currentMediaId` comparisons elsewhere) and `MediaMetadata.MEDIA_TYPE_RADIO_STATION`/`MEDIA_TYPE_FOLDER_RADIO_STATIONS`.
- Tapping a station in the browse tree reaches `onAddMediaItems` (the standard media3 hook for any external controller's `setMediaItem`/`playFromMediaId`-style request) with a bare `MediaItem` carrying just the tapped `mediaId`. This is deliberately **not** routed back through `startPlayback()`/`buildMediaSession()` — releasing and rebuilding the very `MediaLibrarySession` whose callback is currently executing is not something media3 supports safely. Instead, `startPlayback()` was split into `buildMediaSession(streamUrl)` (unchanged, only ever called from top-level entry points: `onCreate`, `onStartCommand` → `startPlayback`) + `applyPlayback(streamUrl, isRetry)` (the actual player/notification/timeshift/retry pipeline, safe to call reentrantly). `onAddMediaItems` resolves the tapped `mediaId` against `cachedStations` (refreshed on every `onGetChildren` call) and, if found, sets `stationName` + calls `applyPlayback()` directly, returning an empty item list so media3 doesn't also try to set it on the player itself.
- `loadBrowsableStations()` and `playFromBrowseTree(mediaId)` are `internal` (not `private`) on `RadioPlaybackService`, same reasoning as `isHlsUrl`/`retryDelayMs`/`isRetryableNetworkError`: media3's own team tests `MediaLibrarySession`/`MediaBrowser` round trips with **instrumentation**, not Robolectric (see their `media3-test-session-current` module, which lives under `src/androidTest`, not `src/test`) — this project deliberately has no `androidTest` (see Testing below), so `RadioPlaybackServiceAutoTest` exercises the real DB → media item → playback pipeline through these two methods directly rather than attempting a full `MediaBrowser` Binder handshake in Robolectric.

## Home screen widget

- `widget/RadioWidget.kt` is a Jetpack Glance (`androidx.glance:glance-appwidget`) `GlanceAppWidget`, registered via `widget/RadioWidgetReceiver.kt` (`GlanceAppWidgetReceiver`) and `res/xml/radio_widget_info.xml` (`android:initialLayout="@layout/widget_loading"` is a required-but-unused placeholder `FrameLayout` — Glance replaces it with the real composable at render time; AppWidgetProviderInfo just needs *some* valid layout resource there). Shows the last-played/current station's name + play-pause state + skip buttons, styled with the same dark palette as `ui/theme/Color.kt` since the whole app (and now the widget) is dark-only by design.
- `ui/playback/WidgetStateStore.kt` (SharedPreferences, same shape as `PlaybackStateStore`/`AlarmStateStore`) is what the widget actually reads — deliberately a separate store from `PlaybackStateStore`, because that one's contract is "last *started* station, cleared on explicit stop" (so it can resume after a process-death restart), whereas the widget wants to keep showing the last station with a play affordance even after the user explicitly stops, not go blank. `RadioPlaybackService.updateWidget()` writes to it and calls `RadioWidget().updateAll(context)` at the end of every `refreshSnapshot()` (i.e. on every player/session state change) — no separate observer/worker needed since the service is already the single source of truth for playback state.
- Button taps are `ActionCallback`s in `widget/WidgetActions.kt`: `TogglePlaybackAction` (stops via `RadioPlaybackService.ACTION_STOP` — a new `onStartCommand` intent-action branch, since previously the only way to stop was `RadioPlaybackService.stopPlayback()` called on a *bound* instance, which the widget's `ActionCallback` isn't) or resumes the last station; `NextStationAction`/`PreviousStationAction` start the neighboring station via `util/StationNavigator.kt` (a pure, plain-JUnit-testable next/prev index calculation over whatever station-list ordering is passed in, same "extract the pure decision logic" pattern as `AlarmScheduler.nextTriggerMillis`) relative to what `WidgetStateStore` says is currently showing, wrapping around at either end. None of these can take a constructor-injected `RadioStationRepository`/`WidgetStateStore` (Android requires a no-arg constructor for `ActionCallback` the same way it does for `Service`/`BroadcastReceiver`), so they resolve `RadioStationRepository.create(context)` themselves, same as `RadioPlaybackService` and `util/AppShortcuts.kt` already do.
- Testing: `RadioWidgetTest` renders the widget's composable (`RadioWidgetContent`, `internal` specifically so a test can call it directly with fixed inputs — the pattern `androidx.glance:glance-appwidget-testing`'s `runGlanceAppWidgetUnitTest` is designed for, per its own doc: "test your individual Glance composable functions in isolation") and asserts on the node tree (`onNode(hasText(...))`, `onNode(hasRunCallbackClickAction<TogglePlaybackAction>())`) rather than going through `RadioWidget.provideGlance()`'s `WidgetStateStore`/`Context` read. `WidgetActionsTest` instead calls each `ActionCallback.onAction()` directly (it's a plain public suspend method) and asserts on the resulting `Intent` via `shadowOf(application).nextStartedService`, the same "drive the real component" style as `AlarmReceiverTest`.
- Both `RadioPlaybackServiceAutoTest` and `WidgetActionsTest`'s next/prev test touch the real `AppDatabase.getDatabase()` singleton (not an isolated in-memory Room DB like every other existing DB test uses) because `RadioStationRepository.create(context)` is hardcoded inside `RadioPlaybackService`/`ActionCallback`s with no injection seam — `AppDatabase`'s companion-object `instance` is cached for the process lifetime (same hazard `StationShareTest` already documents for `FileProvider`'s static per-authority cache), so each of those methods (a) sticks to a single `@Test` method touching the DB and (b) defensively clears all existing rows before seeding, rather than assuming a clean slate.

## Testing

- Unit tests only (`app/src/test`, no `androidTest`); CI's `test` job runs `./gradlew test`. Room DB tests and anything needing a `Context` use Robolectric (`@RunWith(RobolectricTestRunner::class)`, `@Config(sdk = [29])`).
- `StreamRecorder`/`TimeshiftController`/`StreamValidator`/`RadioBrowserApi` tests spin up a local `okhttp3.mockwebserver.MockWebServer` instead of mocking OkHttp, since all four classes construct their own `OkHttpClient` internally (not injected). Any test that touches `org.json` (e.g. `RadioBrowserApiTest`) needs `@RunWith(RobolectricTestRunner::class)` even if it doesn't otherwise need a `Context` — the plain Android SDK stub jar used for non-Robolectric unit tests throws `RuntimeException: ... not mocked` on real `org.json` calls.
- `RadioPlaybackService`'s pure decision logic (`isHlsUrl`, `retryDelayMs`, `isRetryableNetworkError`) is `internal` (not `private`) specifically so it's callable from tests without exercising the full `MediaLibraryService`/ExoPlayer lifecycle; `loadBrowsableStations`/`playFromBrowseTree` (Android Auto browse tree, see above) follow the same pattern and `RadioPlaybackServiceAutoTest` does construct the full service via `Robolectric.buildService(RadioPlaybackService::class.java).create().get()` (a real `ExoPlayer` + `MediaLibrarySession` do come up fine under Robolectric) — it's specifically the `MediaBrowser`-Binder-round-trip part that's avoided, not the service lifecycle itself.
- Glance widget composables (`widget/RadioWidgetContent`) are tested with `androidx.glance:glance-appwidget-testing`'s `runGlanceAppWidgetUnitTest` (`RadioWidgetTest`) — pure node-tree assertions, no rendering, runs fine under Robolectric alongside everything else. Widget `ActionCallback`s (`widget/WidgetActions.kt`) are plain classes with a public suspend `onAction`, so `WidgetActionsTest` just calls that directly rather than going through Glance's click-dispatch machinery.
- ViewModel tests that exercise a suspend function hopping onto the real `Dispatchers.IO` (anything using `StreamValidator`/`RadioBrowserApi`, whose network calls run off the virtual `TestCoroutineScheduler`) must poll for the expected state (`awaitTrue { condition }` wrapped in `withTimeout`, see `DiscoverStationsViewModelTest`) rather than a single `advanceUntilIdle()`, which only fast-forwards the test dispatcher's own queue and won't wait for real background-thread work to finish.
- Even fully-virtual `MainViewModelTest`-style tests (`StandardTestDispatcher` + in-memory Room, no real dispatcher crossing) need a *content*-based poll condition, not a count-based one, when asserting after a `viewModelScope.launch`-fired update that doesn't change the collection size (e.g. `toggleFavorite`) — `waitForStationsCount(viewModel, sameSizeAsBefore)`'s while-loop condition is already true before the launched coroutine runs, so it returns without ever calling `advanceUntilIdle()`, and the assertion races the update. Poll the actual field that's expected to change instead (see `waitUntil { ... }`).
- `IconStorageTest` also opts into `@GraphicsMode(GraphicsMode.Mode.NATIVE)` (like the screenshot tests below) even though it captures no screenshot — plain/legacy Robolectric fakes `BitmapFactory` decode results, so asserting real downsampled pixel dimensions after `IconStorage.saveImage()` needs the same real-Skia-backed decode path. `AddStationViewModelTest`'s save-triggers-file-cleanup test follows `DiscoverStationsViewModelTest`'s `awaitTrue` real-time-poll pattern (see below) since `StreamValidator.isReachable()` is on the save path; it polls the actual filesystem state (`!oldIconFile.exists()`), not `!uiState.value.isSaving`, since `isSaving` starts `false` and would satisfy an `awaitTrue` before the save coroutine ever runs.
- **Never call a suspend function inside `awaitTrue`'s `poll: () -> Boolean` lambda** (or any tight `while (!cond) { advanceUntilIdle() }` loop) — the poll call itself would suspend on a real background thread (Room's executor, OkHttp), and since nothing else is driving the virtual scheduler forward while *you're* the one suspended, the coroutine you're waiting on (e.g. `save()`) never gets to progress either: a real deadlock, only broken by `withTimeout` firing after the full virtual-time budget. When there's no synchronous side effect to poll (contrast the icon-cleanup test's `!oldIconFile.exists()` above), poll a plain `StateFlow` field twice in sequence instead — first for the transient "in progress" value, then for it flipping back (`awaitTrue { uiState.value.isSaving }` then `awaitTrue { !uiState.value.isSaving }`) — rather than reading the DB directly from inside the poll (see `AddStationViewModelTest`'s genre-persists-on-save tests).
- **`awaitTrue`'s `advanceUntilIdle()`-busy-loop stops being reliable once a background chain does more than one real-dispatcher hop back-to-back** (`DiscoverStationsViewModelTest`'s favicon-download test: network IO → `withContext(Dispatchers.IO)` bitmap decode/file write → two sequential Room calls) — repeated `advanceUntilIdle()` calls with no actual elapsed wall-clock time between them can race ahead of a still-in-flight later hop, since nothing forces real progress on the real thread in between; a `delay()`-based retry loop doesn't fix this either, since `withTimeout`/`delay` under `runTest` operate on *virtual* time by default and never actually sleep in real time (confirmed by that test's `TimeoutCancellationException: ... 5s of _virtual_ time` when a `delay(20)` retry loop was tried — see git history). The fix that actually works: a plain `while` loop bounded by real `System.currentTimeMillis()`, reading state through `kotlinx.coroutines.runBlocking { ... }` (a genuinely separate blocking call to the real DB, not tied to `testScheduler` — safe here, unlike the raw-suspend-in-`poll()` deadlock above, precisely *because* it doesn't participate in the virtual scheduler's cooperative machinery) with a real `Thread.sleep(20)` between attempts so the background thread(s) actually get wall-clock time to finish.
- Screenshot tests (e.g. `StationItemScreenshotTest`) use Roborazzi over Robolectric Native Graphics (`@GraphicsMode(GraphicsMode.Mode.NATIVE)`, still `@RunWith(RobolectricTestRunner::class)` like every other test here) to catch pixel-level regressions a plain unit test can't (a shadow clipped into a rectangle, text invisible against its background) — capture with the plain `com.github.takahirom.roborazzi.captureRoboImage { content }` lambda from `roborazzi-compose`, not `createComposeRule()`/`createAndroidComposeRule<ComponentActivity>()`: this project's `AndroidManifest.xml` has no `ComponentActivity` launcher entry (and merging one in via a `src/test/AndroidManifest.xml` override still isn't enough — Robolectric's shadow `PackageManager` refused to resolve it, see git history for the abandoned attempt), whereas the lambda form registers Roborazzi's own `RoborazziActivity` into the shadow `PackageManager` at runtime (`registerRoborazziActivityToRobolectricIfNeeded()`) and needs no manifest entry or Activity-hosting rule at all. Reference images are committed to `app/src/test/screenshots/` (`roborazzi { outputDir.set(file("src/test/screenshots")) }` in `app/build.gradle` — the Roborazzi default of `build/outputs/roborazzi` would be invisible to git and to CI, since `build/` is gitignored). `./gradlew recordRoborazziDebug` re-records the baseline after an intentional visual change (review the diff before committing); `./gradlew verifyRoborazziDebug` (wired into CI's `test` job, after `./gradlew test`) fails the build on any pixel diff; plain `./gradlew test` runs the same tests but doesn't fail on a mismatch, so screenshot regressions only get caught by the dedicated verify step.
